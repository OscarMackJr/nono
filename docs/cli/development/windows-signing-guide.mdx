# Windows Signing Guide

This guide covers the Authenticode signing setup required to produce signed Windows release artifacts.

Signing runs only in CI for tag or release builds. Local developer builds are never signed.
This is intentional: the signing secrets are CI-only and the workflow is designed to fail
closed on any signing or verification error (D-10, D-13).

## CI signing: Azure Trusted Signing (keyless)

> **D-02 remediation (quick-task `260603-i31`).** CI Authenticode signing migrated from a
> self-signed/base64-PFX cert to **Azure Trusted Signing** — a Microsoft-managed, publicly
> trusted, **keyless** signing service. There is **no PFX and no private key in CI**; the
> runner authenticates to Azure via GitHub OIDC and the signing key lives in Microsoft's HSM.
> Full one-time setup (Azure account, certificate profile, OIDC identity, RBAC) is in the
> cookbook: `.planning/quick/260603-i31-cosign-sigstore-signing-authority/AZURE-TRUSTED-SIGNING-COOKBOOK.md`.

The release workflow needs these **repository secrets** (only Azure identity IDs — no key material):

| Secret | Description |
|--------|-------------|
| `AZURE_CLIENT_ID` | App registration (service principal) client ID, federated to GitHub OIDC |
| `AZURE_TENANT_ID` | Entra tenant ID |
| `AZURE_SUBSCRIPTION_ID` | Subscription holding the Trusted Signing account |

…and these **repository variables** (non-secret):

| Variable | Description | Example |
|----------|-------------|---------|
| `TRUSTED_SIGNING_ENDPOINT` | Region endpoint of the Trusted Signing account | `https://eus.codesigning.azure.net` |
| `TRUSTED_SIGNING_ACCOUNT` | Trusted Signing account name | `nono-trusted-signing` |
| `TRUSTED_SIGNING_PROFILE` | Public Trust certificate profile name | `nono-public` |

The service principal must hold the **"Trusted Signing Certificate Profile Signer"** role on the
account. Set secrets/variables at `Settings -> Secrets and variables -> Actions`.

If the OIDC credentials or Trusted Signing account are absent/misconfigured, the `azure/login`
step (or the signing step) fails and **no unsigned artifact is uploaded** — fail-closed is
preserved (D-13). Unsigned Windows artifacts are never releasable.

> **Legacy (no longer used by `release.yml`):** the previous CI path used base64-PFX repository
> secrets `WINDOWS_SIGNING_CERT` / `WINDOWS_SIGNING_CERT_PASSWORD` consumed by
> `scripts/sign-windows-artifacts.ps1`. Those are retained only for the **local POC signing**
> flow below; the CI workflow no longer references them.

## What gets signed

The release workflow signs these artifacts before uploading them:

- `nono.exe` — the compiled binary
- `nono-vX.Y.Z-x86_64-pc-windows-msvc-machine.msi` — the per-machine MSI installer
- `nono-vX.Y.Z-x86_64-pc-windows-msvc-user.msi` — the per-user MSI installer

The `.zip` artifact (`nono-vX.Y.Z-x86_64-pc-windows-msvc.zip`) contains the already-signed
`nono.exe` and is assembled after signing. The release workflow also publishes the signed raw
`nono.exe` as a release asset so signature verification does not require zip extraction.

## Signing implementation

> **CI signs via Azure Trusted Signing** (the `azure/trusted-signing-action` steps in
> `release.yml`), not the PFX script below. The `signtool` + base64-PFX flow described in this
> section is now the **local POC signing** primitive (`scripts/sign-windows-artifacts.ps1`,
> invoked by `sign-poc-local.ps1`); it is no longer the CI path. `release.yml` keeps the same
> signing **order** (sign binaries → package MSI → sign MSI wrappers → verify) and the same
> Authenticode / MSI-payload / zip verify gates, which are signing-primitive-agnostic.

Local POC signing is performed by `scripts/sign-windows-artifacts.ps1`. The script:

1. Decodes `WINDOWS_SIGNING_CERT` from base64 to a temporary PFX file.
2. Imports the certificate into `Cert:\CurrentUser\My` for the duration of the job.
3. Calls `signtool.exe sign /fd sha256 /sha1 <thumbprint> /tr http://timestamp.digicert.com /td sha256 <artifact>` for each artifact.
4. Calls `signtool.exe verify /pa /tw <artifact>` for each artifact. The `/tw` flag treats a missing timestamp as a failure, ensuring every released artifact has a durable timestamp (D-12).
5. Removes the certificate from the store and deletes the temporary PFX file in all exit paths (including errors).

The script requires `signtool.exe` on `PATH`, which is available on GitHub Actions `windows-latest` runners via the pre-installed Windows SDK.

### Timestamping

The script uses RFC 3161 timestamping via `/tr` and `/td sha256`. This is the current
Microsoft-recommended mode for SHA-256 signed binaries. The DigiCert endpoint
(`http://timestamp.digicert.com`) supports RFC 3161 when addressed via `/tr`.

D-12 (locked, **local POC path**): DigiCert timestamping, SHA-256 only. (CI Trusted Signing uses
the Microsoft RFC 3161 endpoint `http://timestamp.acs.microsoft.com`, SHA-256.)
D-14 (locked, **local POC path**): `signtool.exe` is the only signing primitive for this script.
No custom PKCS#7 logic or alternative tooling is permitted. (CI signs via the
`azure/trusted-signing-action`, which wraps `signtool` + the Trusted Signing dlib.)

## Verifying signatures

To verify an Authenticode signature on a downloaded artifact:

```powershell
$sig = Get-AuthenticodeSignature -FilePath "nono.exe"
$sig.Status
# Expected: Valid

$sig.SignerCertificate.Subject
# Expected: CN=<your organization name>, ...
```

For MSI artifacts, pass the MSI path:

```powershell
Get-AuthenticodeSignature -FilePath "nono-v0.2.0-x86_64-pc-windows-msvc-machine.msi" |
    Select-Object Status, SignerCertificate
```

`Status` must be `Valid`. Any other status (`NotSigned`, `HashMismatch`, `UnknownError`) means
the artifact is not distributable.

The release workflow checks these statuses automatically before upload for:

- the signed `nono.exe`
- the machine MSI
- the user MSI
- the extracted `nono.exe` inside the zip payload

## Generating a test certificate

To test the signing workflow without a production code-signing certificate, generate a self-signed certificate:

```powershell
$cert = New-SelfSignedCertificate `
    -Subject "CN=nono Test Signing" `
    -CertStoreLocation "Cert:\CurrentUser\My" `
    -Type CodeSigningCert `
    -HashAlgorithm SHA256

$password = ConvertTo-SecureString -String "TestPassword123!" -Force -AsPlainText
Export-PfxCertificate -Cert $cert -FilePath "nono-test.pfx" -Password $password

$base64 = [Convert]::ToBase64String([IO.File]::ReadAllBytes("nono-test.pfx"))
Write-Host $base64

Remove-Item "nono-test.pfx" -Force
Remove-Item -Path $cert.PSPath -Force
```

Do not use self-signed certificates in production releases. Production releases require a
certificate from a trusted CA enrolled in the Microsoft Trusted Root Program.

Note that `signtool verify /pa /tw` will warn or fail for self-signed certificates because
they are not trusted by the default authentication policy (`/pa`). Use
`signtool verify /pa /tw /r <thumbprint>` or accept the expected failure output when running
local tests with self-signed certs.

## Runtime symptom: unsigned install cannot spawn the broker

If you run an **unsigned** install (for example a locally built MSI deployed to
`C:\Program Files\nono\`), `nono run` fails when it tries to spawn the Windows
Low-IL broker with:

```text
Trust verification failed for C:\Program Files\nono\nono.exe:
nono.exe Authenticode status is Unsigned (expected Valid).
Self-trust-anchor unavailable; refusing to spawn broker.
```

This is **fail-secure behavior working as designed**, not a bug (Phase 32, D-32-12). The
broker-spawn gate (`verify_broker_authenticode` in
`crates/nono-cli/src/exec_strategy_windows/launch.rs`) implements a *self-trust-anchor*:
`nono.exe` reads its own Authenticode signer subject + thumbprint and requires the sibling
`nono-shell-broker.exe` to match exactly. That anchor only exists when `nono.exe`'s own
signature is `Valid`, so an unsigned production-layout install has no anchor to extract and
the gate fail-closes. There is **no env-var or CLI escape hatch** in production layout
(D-32-12).

There are exactly two supported ways to make `nono run` succeed:

1. **Install a signed release (production fix).** Sign `nono.exe` *and*
   `nono-shell-broker.exe` with the same certificate so their signer subject + thumbprint
   match. In CI this happens automatically once the `WINDOWS_SIGNING_CERT` /
   `WINDOWS_SIGNING_CERT_PASSWORD` repository secrets are set (see
   [Required GitHub Actions secrets](#required-github-actions-secrets)) — the release
   workflow signs both binaries and both MSIs before upload. Shipped MSIs are unsigned only
   when those secrets are absent.

2. **Run from a dev-build layout (development).** The gate is intentionally skipped when
   `nono.exe` runs from a `\target\release\` or `\target\debug\` path (`is_dev_build_layout`).
   Build and run the dev-layout binaries directly — no signing, no admin, no trust-store
   changes:

   ```powershell
   cargo build --release -p nono-cli -p nono-shell-broker
   # then invoke the dev-layout binary (broker is found as its sibling):
   .\target\release\nono.exe run --profile claude-code -- claude --version
   ```

For a **single developer machine**, prefer option 2 — it needs no cert and no trust-store
changes. Signing an unsigned install in place with a self-signed cert is not a *production*
path: because the gate uses `WinVerifyTrust` with full chain trust, the self-signed cert must
also be imported into the machine's Trusted Root and Trusted Publisher stores, which broadens
that machine's code-trust surface.

For a **controlled internal POC across a few machines you own** (where running each from a
`target\release` layout is impractical), that trust-store import is an acceptable, reversible
tradeoff. See [Local POC signing](#local-poc-signing-self-signed-internal-use) below, which
wraps the whole flow in `scripts/sign-poc-local.ps1`.

## Local POC signing (self-signed, internal use)

> **POC-only. Not for external distribution.** This path uses a self-signed certificate that
> is trusted only on machines where you explicitly import it. It broadens those machines'
> code-trust surface to anything signed by that cert, so use it **only on machines you
> control** (e.g. an internal admin team's boxes). It is fully reversible. For releases to
> anyone else, use a commercial CA code-signing cert or Azure Trusted Signing and the CI
> `release.yml` path above. **You do not need DigiCert as a CA** — the only DigiCert
> dependency anywhere here is the free public timestamp server.

`scripts/sign-poc-local.ps1` performs the entire local POC signing flow in one command:

```powershell
pwsh -File scripts/sign-poc-local.ps1
# or scope/cert options:
pwsh -File scripts/sign-poc-local.ps1 -Scope machine -VersionTag v0.57.3
pwsh -File scripts/sign-poc-local.ps1 -Thumbprint <existing-cert-thumbprint>
```

It runs, in this order:

1. Discovers `signtool.exe` (newest `Windows Kits\10\bin\<ver>\x64`).
2. Creates a self-signed `CodeSigningCert` in `Cert:\CurrentUser\My` (SHA-256, +2y), or
   reuses one via `-Thumbprint`.
3. **Signs the EXEs first** — `nono.exe`, `nono-shell-broker.exe`, and (machine scope only,
   if present) `nono-wfp-service.exe`.
4. **Rebuilds the MSIs** via `build-windows-msi.ps1` so the cab embeds the *signed* EXEs.
5. **Signs the MSIs.**
6. Exports the **public** certificate (no private key) to
   `dist/windows/nono-poc-signing.cer` for distribution.

**Why the EXE-then-MSI order is mandatory:** the MSI embeds *copies* of the EXEs, and the
broker gate checks the **installed** `nono.exe`'s signature — not the MSI's. Signing the MSI
alone leaves unsigned EXEs inside it and the broker still fail-closes. The script enforces the
order; `-SkipMsiRebuild` re-signs existing artifacts only and warns about this.

**The private key never leaves the build machine.** Distribute only the public `.cer` and the
signed MSI. On **each** target machine, an admin runs once:

```powershell
Import-Certificate -FilePath nono-poc-signing.cer -CertStoreLocation Cert:\LocalMachine\Root
Import-Certificate -FilePath nono-poc-signing.cer -CertStoreLocation Cert:\LocalMachine\TrustedPublisher
msiexec /i nono-v0.57.3-x86_64-pc-windows-msvc-machine.msi
(Get-AuthenticodeSignature 'C:\Program Files\nono\nono.exe').Status   # -> Valid
nono run --profile claude-code --allow-cwd -- claude --version
```

Once the cert is in **Root + TrustedPublisher**, `nono.exe` verifies as `Valid`, the broker
self-trust-anchor exists, and the gate passes.

**Reversibility:** to undo on a machine, delete the cert from `Cert:\LocalMachine\Root` and
`Cert:\LocalMachine\TrustedPublisher` and uninstall the MSI.

> Running the helper against the default `-OutputDir dist/windows` regenerates the tracked
> `dist/windows/*.wxs` reference snapshots (a side effect of `build-windows-msi.ps1`). Revert
> with `git checkout -- dist/windows/nono-machine.wxs dist/windows/nono-user.wxs` if you do
> not want that churn.

## Production signing path — Azure Trusted Signing (current)

`release.yml` signs with **Azure Trusted Signing** (keyless, Microsoft-trusted root). This is
the production path; it does **not** use a PFX or the `WINDOWS_SIGNING_CERT` secret. To make CI
produce signed releases:

1. Create + identity-validate an **Azure Trusted Signing account** and a **Public Trust**
   certificate profile. **Validate eligibility first** — Public Trust requires Microsoft
   identity validation (legal-entity / individual verification).
2. Create an Entra app registration, federate it to GitHub OIDC, and grant it the
   **"Trusted Signing Certificate Profile Signer"** role on the account.
3. Set the repository secrets `AZURE_CLIENT_ID` / `AZURE_TENANT_ID` / `AZURE_SUBSCRIPTION_ID`
   and the repository variables `TRUSTED_SIGNING_ENDPOINT` / `TRUSTED_SIGNING_ACCOUNT` /
   `TRUSTED_SIGNING_PROFILE` (see [CI signing](#ci-signing-azure-trusted-signing-keyless) above).
4. No further code change is required — `release.yml`'s `azure/login` + `azure/trusted-signing-action`
   steps consume them. The signing order and all verify gates are unchanged.

Full step-by-step (Azure CLI commands, RBAC, OIDC federation, the exact `release.yml` wiring,
cost, and the WFP kernel-driver EV/attestation caveat) is in the cookbook:
`.planning/quick/260603-i31-cosign-sigstore-signing-authority/AZURE-TRUSTED-SIGNING-COOKBOOK.md`.

### Alternative: commercial PFX cert (OV/EV from a CA)

If you would rather use a purchased code-signing certificate (e.g. DigiCert, Sectigo) instead of
Trusted Signing, revert the two `release.yml` signing steps to the PFX flow
(`scripts/sign-windows-artifacts.ps1 -CertBase64 … -CertPassword …`) and set:

1. Obtain a code-signing cert from a CA in the Microsoft Trusted Root Program; export as a
   password-protected PFX (certificate + private key).
2. Base64-encode it: `[Convert]::ToBase64String([IO.File]::ReadAllBytes("cert.pfx"))`.
3. Set `WINDOWS_SIGNING_CERT` (base64) and `WINDOWS_SIGNING_CERT_PASSWORD` repository secrets.

The PFX signing primitive (`signtool /sha1 <thumbprint>`) is identical whether the cert is
self-signed or CA-issued; only the trust chain differs. Note an OV cert still warms up SmartScreen
reputation gradually; an EV cert (or Trusted Signing) avoids the warm-up. Kernel drivers
(`nono-wfp-driver.sys`) require the separate EV + Microsoft Partner Center attestation/WHQL flow
regardless of which user-mode path you choose.

## Fresh CI cert: generation, encoding, and trust redistribution

D-53-02: each v2.x release uses a fresh self-signed cert (provenance born in/for CI),
NOT a re-export of a previous POC cert. Consequence: the new cert's public `.cer` MUST
be redistributed to every trusted machine before they accept the new signed binaries.
Old installs' trust does not carry over automatically.

**Step 1: Generate the fresh cert (elevated PowerShell on the operator's machine):**

Use the cert-gen block from `sign-poc-local.ps1` (lines 137-144): run
`New-SelfSignedCertificate` with subject `CN=nono POC Signing v0.57.4` (or similar
version-tagged subject to distinguish certs across releases), type CodeSigningCert,
HashAlgorithm SHA256, NotAfter two years from now. The cert lands in
`Cert:\CurrentUser\My`.

```powershell
$cert = New-SelfSignedCertificate `
    -Subject "CN=nono POC Signing v0.57.4" `
    -CertStoreLocation "Cert:\CurrentUser\My" `
    -Type CodeSigningCert `
    -KeyUsage DigitalSignature `
    -HashAlgorithm SHA256 `
    -NotAfter (Get-Date).AddYears(2)
Write-Host "Thumbprint: $($cert.Thumbprint)"
```

**Step 2: Export the PFX (private + public key, password-protected):**

```powershell
$password = ConvertTo-SecureString "YourStrongPassword" -AsPlainText -Force
Export-PfxCertificate -Cert $cert -FilePath "nono-v0.57.4-ci.pfx" -Password $password
```

**Step 3: Base64-encode the PFX for the secret:**

```powershell
$b64 = [Convert]::ToBase64String([IO.File]::ReadAllBytes("nono-v0.57.4-ci.pfx"))
$b64 | Out-File "nono-v0.57.4-ci.pfx.b64.txt"
# Content of that txt is the WINDOWS_SIGNING_CERT value
```

**Step 4: Set GitHub repository secrets:**

Navigate to `Settings -> Secrets and variables -> Actions`. Create or update:

- `WINDOWS_SIGNING_CERT` — the base64 PFX value from Step 3.
- `WINDOWS_SIGNING_CERT_PASSWORD` — the plaintext password from Step 2.

The PFX file must be securely deleted after encoding:

```powershell
Remove-Item "nono-v0.57.4-ci.pfx" -Force
```

**Step 5: Export the public `.cer` for trust redistribution (NO private key):**

```powershell
Export-Certificate -Cert $cert -FilePath "nono-v0.57.4-poc.cer" -Type CERT
```

Distribute this `.cer` to every machine that needs to accept v0.57.4-signed binaries.
On each target machine (admin, one-time):

```powershell
Import-Certificate -FilePath "nono-v0.57.4-poc.cer" -CertStoreLocation Cert:\LocalMachine\Root
Import-Certificate -FilePath "nono-v0.57.4-poc.cer" -CertStoreLocation Cert:\LocalMachine\TrustedPublisher
```

After import, `(Get-AuthenticodeSignature 'C:\Program Files\nono\nono.exe').Status` must
report `Valid`. Without this step, the broker self-trust-anchor gate (D-32-12) will
fail-close on the new binaries even on machines that accepted previous signed releases.
**Old cert trust does not carry over automatically** — each new cert requires its own
redistribution round.

**Step 6: Push main, then cut the dual tags:**

Local `main` must be pushed before cutting the semver tag (the release workflow checks
out the tag ref — a tag on an unpushed commit is unreachable to CI):

```powershell
git push origin main
git tag v0.57.4
git tag v2.8
git push origin v0.57.4 v2.8
```

Only `v0.57.4` fires the release workflow (the `v*.*.*` trigger); `v2.8` is a non-firing
milestone marker (two-segment tag does not match the glob).
