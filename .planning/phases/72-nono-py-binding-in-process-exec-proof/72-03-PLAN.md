    ## E1 — Executable and Interpreter Paths

    Prose + invariants:
    - The launch profile must enumerate every binary the engine invokes (python.exe, node.exe, etc.)
      via --allow <exe-dir> entries.
    - The exe/interpreter coverage gate is fail-secure: an uncovered binary is refused, never
      silently degraded to partial confinement.
    - Paths must be absolute. The PowerShell-to-C:\ relative-write trap (spike 003 finding) proves
      engines do not uniformly inherit the launcher CWD.

    ## E2 — Ownable Launch Command

    Prose + invariants:
    - nono (or the daemon in Phase 74) must be the parent process — it owns the spawn, not the IDE
      or a third-party launcher.
    - If a third party owns the spawn (IDE-embedded engine, already-running process), fall back to
      E5 (pre-exec interception) or adopt with the documented weaker guarantees (Phase 73).

    ## E3 — Absolute Writable Workspace

    Prose + invariants:
    - All workspace grants are absolute paths — never relative to the launcher CWD.
    - The workspace must be user-owned (not elevated/admin-owned): an admin-owned dir defeats the
      DACL/label grant (no WRITE_DAC, R-B3 — nono emits a clear diagnostic).
    - Confinement guarantees: NO_WRITE_UP to unauthorized paths, deny-network-unless-granted.

    ## E4 — Network Identity

    Prose + invariants:
    - Each confined agent is assigned a per-agent network identity (AppContainer package SID on the
      broker no-PTY arm) for WFP scoping in Phase 75.
    - The identity is established at spawn time, not mid-run.

    ## E5 — Pre-Exec Interception Point (Optional)

    Prose + invariants:
    - E5 applies ONLY when nono cannot be the parent (IDE-embedded engine, already-running process).
    - Built consumers: Claude PreToolUse hook (shipped), Copilot JSON-RPC hooks (Phase 75).
    - Forward-compat note: The E5 hook slot is the integration point for the zt-infra.org control
      plane (see "Forward-Compat: zt-infra.org Integration" below).

    ## Forward-Compat: zt-infra.org Integration

    This section documents how the E5 slot maps to the zt-infra.org fail-closed agent security
    control plane. The integration itself is a FUTURE phase. The design documented here ensures
    nono does not paint itself into a corner.

    zt-infra.org role: fail-closed agent security control plane (POST /actions allow/deny + crypto
    audit); lists nono as its execution sandbox; supports LangGraph/LangChain/MCP over HTTP;
    Linux-container/cloud runtime (https://www.zt-infra.org/).

    Mapping:
      E5 pre-exec interception point (FUTURE via zt-infra):
        1. agent submits tool call intent to zt-infra control plane via POST /actions
        2. control plane evaluates policy + records crypto audit trail
        3. decision: allow / deny
        4. deny => skip exec, fail-closed (nono's confinement still enforces OS-level boundary)
        5. allow => exec proceeds; nono enforces OS confinement UNDERNEATH the policy decision

    nono's role: enforce OS confinement regardless of the E5 decision — nono is the execution
    sandbox under the control plane, not a replacement for it. Even if E5 allows an action,
    nono's Low-IL/AppContainer/WFP enforcement applies.

    ## Implementation Notes per Platform

    ### Windows (Phase 72+)

    Shape A (confined_run): nono-py confined_run() spawns nono.exe run --profile … -- <exe> <args>
    via subprocess. The whole python process runs confined from the start. This is the preferred
    default for LangChain PythonREPLTool (in-process exec() is confined because the PROCESS
    is confined).

    Shape B (confine — born-confined self-re-exec): nono-py confine() re-execs the current python
    process through nono.exe as the FIRST operation in main() before any privileged handle is
    opened. The agent process is BORN CONFINED at its own entrypoint. The NONO_ALREADY_CONFINED
    env var prevents infinite re-exec. confine() MUST be called before any file handle, socket,
    registry key, or other privileged object is acquired — ORDERING IS THE INVARIANT.

    Note: Windows Sandbox::apply() (nono lib) is preview/advisory-only — it validates the cap set
    but does not enforce OS restrictions on the running process. Binding-level confinement on
    Windows ALWAYS routes through nono.exe (the broker). This is by design (D-03).

    ### Linux (Landlock)

    Sandbox::apply() on the CURRENT process applies Landlock restrictions in-process. sandboxed_exec()
    forks + applies + exec. confine() on Linux can call Sandbox::apply() directly (no re-exec needed;
    enforcement is genuinely in-process). Shape B on Linux is simpler and more direct than on Windows.

    ### macOS (Seatbelt)

    Sandbox::apply() via sandbox_init() applies the Seatbelt profile to the current process.
    Same in-process enforcement as Linux Landlock — Shape B is direct on macOS.
