<objective>
Fix the two macOS resource-limit enforcement bugs surfaced as the Phase 65 gate-65-A "A5" finding:

1. --max-processes is a silent no-op. Both the Direct path (supervisor_macos.rs install_pre_exec) and the Supervised path (exec_strategy.rs ForkResult::Child arm) never call setrlimit(RLIMIT_NPROC) — they only log a warning. Replace with real libc::setrlimit(libc::RLIMIT_NPROC, baseline+N) fail-closed enforcement.

2. --timeout watchdog fires but kills the wrong process group. The child inherits the parent's process group; kill(-pgrp, SIGKILL) misses or kills the parent. Fix by calling setpgid(0,0) in the child arm immediately after fork, making the child its own process-group leader so the existing watchdog kill(-child_pgrp, SIGKILL) targets only the agent tree.

Purpose: These are nono supervisor bugs on macOS (RESL-MAC-01, RESL-MAC-02) — not OS limitations, not test-gating workarounds. Real enforcement on a real host.

Output:
- uid_process_count() helper in supervisor_macos.rs (parent-side sysctl KERN_PROC_UID baseline)
- MacosResourceLimits gains baseline_uid_count: u64 field (Copy, pre-fork, passed into pre_exec closure)
- Direct path fix: install_pre_exec calls libc::setrlimit(RLIMIT_NPROC) with fail-closed Err return
- Supervised path fix: ForkResult::Child arm adds setpgid(0,0) + real RLIMIT_NPROC enforcement with MSG_RLIMIT_NPROC_FAIL const + _exit(126)
- D-09 bonus: one macos_memory_limit_kills_at_rlimit_as env-gated test added to resl_nix_macos.rs
- Both RESL gated tests pass on Oscars-MacBook-Pro with NONO_RESL_HOST_VALIDATED=1
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| parent → child (fork) | Parent computes baseline_uid_count and resource limit values; child inherits them via Copy-captured integers in the child arm; no IPC, no deserialization, but the Copy values must be computed correctly |
| child arm → kernel (setrlimit) | setrlimit call applies resource limits to the child process; failure must be detected and cause _exit(126), not silent continuation |
| parent → kernel (sysctl KERN_PROC_UID) | sysctl return value must be checked; failure must propagate as Err before fork; result is used to compute RLIMIT_NPROC soft limit |
| supervisor watchdog → agent process group | kill(-child_pgrp, SIGKILL) targets the group identified by getpgid(child); incorrect group membership can cause wrong-group kill or kill miss |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-68-01 | Elevation of Privilege | Agent process: RLIMIT_NPROC no-op allows unbounded fork() | mitigate | Replace warn! with libc::setrlimit(RLIMIT_NPROC, baseline+N) + _exit(126) on setrlimit failure in both Direct and Supervised paths (D-02, D-07) |
| T-68-02 | Denial of Service | Watchdog kills wrong process group (parent's pgrp instead of agent pgrp) | mitigate | setpgid(0,0) in child arm before sandbox apply makes child its own pgrp leader; WR-04 skip-on-getpgid-failure preserved (D-04, D-06) |
| T-68-03 | Tampering | Baseline count race: new UID processes spawn between sysctl read and setrlimit apply | accept | Documented/accepted per D-03; RLIMIT_NPROC is UID-wide on macOS — inherently racy by design; baseline+N strategy compensates; no practical fix exists without kernel-level true-descendant scoping |
| T-68-04 | Tampering | RLIMIT_NPROC raise above hard limit returns EPERM → silent limit skip if not fail-closed | mitigate | Read existing hard limit via libc::getrlimit before setrlimit; cap rlim_cur and rlim_max at min(target, existing_hard); fail-closed _exit(126) if setrlimit still fails (Pitfall 3) |
| T-68-05 | Elevation of Privilege | Parent-side sysctl failure silently falls back to baseline=0 (trivially low limit) | mitigate | uid_process_count() returns Err on sysctl failure; MacosResourceLimits::new() propagates ? fail-closed; run aborted before fork with NonoError::SandboxInit (D-07) |
| T-68-06 | Tampering | format!() or heap allocation in CR-01 child arm could cause undefined behavior if allocator is in inconsistent state after fork | mitigate | All child-arm error messages are const MSG_*: &[u8] byte string literals; no String, Vec, format! — enforced by cr_01_no_format_macro_in_post_fork_child_branch test |
