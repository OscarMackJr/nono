<objective>
Wire per-agent WFP egress filtering into the daemon launch/reap lifecycle (SUPP-02).

Purpose: Every confined agent launched through the daemon gets its own WFP network-egress filter keyed to its unique AppContainer package SID (E4 identity). The elevated `nono-wfp-service` enforces it; the USER daemon is a least-priv client that sends activate/deactivate requests over the existing control pipe. This satisfies the per-agent network isolation requirement (SC2) and the D-05 fail-secure gate.

Output: `wfp_filter_add` / `wfp_filter_remove` helpers in `launch.rs`; D-05 gate in `launch_agent`; WFP deactivation in the reap task in `reap.rs`. Unit tests covering add-at-launch, remove-at-reap, D-05 fail-secure (service absent + scoped), and D-05 pass-through (service absent + not scoped).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| daemon → nono-wfp-service pipe | Daemon (current user, Medium IL) sends JSON requests to the elevated service. The service SDDL allows IU read/write so the request can be sent; the service validates the SID string via ConvertStringSidToSidW before building any SDDL. |
| launch_agent → suspended child | The agent process is created suspended; its thread is only resumed after WFP filter is installed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-75-01-01 | Tampering | wfp_filter_add: session_sid field | mitigate | The package_sid is minted by the daemon's own CreateAppContainerProfile call (not from wire input); it is never derived from operator-supplied string. The WFP service validates via ConvertStringSidToSidW before SDDL construction. |
| T-75-01-02 | Elevation of Privilege | Stale WFP filter after daemon crash (pitfall 6) | mitigate | Deterministic rule names ("nono-agent-{tenant_id}") let the nono-wfp-service startup sweep identify and remove orphaned filters. Per-agent filters use stable GUIDs derived from rule names, not random GUIDs. Verify at SC2 UAT that the sweep reclaims filters. |
| T-75-01-03 | Denial of Service | D-05 fail-secure gate aborts launch when WFP service absent | accept | This is by design (fail-secure). Profiles that do not declare network scoping are unaffected. Users must install nono-wfp-service to use network-scoped profiles. |
| T-75-01-04 | Elevation of Privilege | Agent starts before WFP filter installed (pitfall 3) | mitigate | wfp_filter_add call is positioned BEFORE ResumeThread in launch_agent (between steps 6 and 7a). Gate is enforced: if wfp_filter_add fails → TerminateProcess → return Err. Agent thread is never resumed without the filter. |
| T-75-01-SC | Tampering | npm/pip/cargo installs | mitigate | No new external packages installed (Phase 75 is zero-new-dep). Slopcheck: not applicable. |
</threat_model>
