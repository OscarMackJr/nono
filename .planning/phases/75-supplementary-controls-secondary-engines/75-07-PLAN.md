<objective>
Fix GAP-75-B (PRIORITY): daemon-launched agents are capability-less. `handle_launch` in control_loop.rs passes `CapabilitySet::new()` (empty) to `launch_agent`, and `launch_agent` never applies package-SID DACL grants. A confined agent therefore has zero filesystem access beyond the AppContainer default (System32), causing real engines to die on startup (`python312.dll not installed`; `claude.exe` / `node.exe` would fail identically).

Purpose: Wire the daemon launch path to match the per-invocation `nono run` path: build a real CapabilitySet from the named profile, construct a minimal `WindowsFilesystemPolicy` covering the engine runtime + system dirs + per-tenant workspace, apply package-SID DACL grants (write+traverse on workspace, read+traverse on read-only paths) BEFORE ResumeThread, store the grants in AgentTenant for revocation on reap.

This is the `launch_agent` ↔ `create_process_containment` dedup carry-forward from Phase 74.

Output: Modified control_loop.rs (caps + workspace derivation), modified launch.rs (DACL grant application + new DaemonDaclGuard), modified reap.rs (dacl_guard field + Drop revocation). Three new unit tests for: caps non-empty for known profile; grant-failure terminates process (fail-secure); reap revokes grants. One `autonomous: false` live claude.exe coverage proof task.
</objective>

Key insight for cap building in daemon context:
The daemon binary (`nono-agentd`) loads `agent_daemon` via `#[path]` and does NOT have access to `crate::capability_ext`, `crate::policy`, or `crate::profile`. The `EMBEDDED_POLICY_JSON` is available. The daemon must build its CapabilitySet from raw JSON parsing — the same approach `is_known_profile` uses, but extending it to extract exe paths and interpreter paths.

Specifically for the daemon's CapabilitySet, the goals are:
1. Grant READ on the resolved engine exe's parent directory (the engine runtime dir, e.g. C:\Users\OMack\.local\bin for claude.exe)
2. Grant READ on each `windows_interpreters` entry's parent directory (e.g. C:\Users\OMack\AppData\Local\Programs\Python\Python312\ for python.exe)
3. Grant READ on Windows system dirs: %SystemRoot% (C:\Windows), %SystemRoot%\System32, %SystemRoot%\SysWOW64 — mandatory for CLR/PE loader (re-assert the 0xFFFF0000 gotcha from Phase 58)
4. Grant READ+WRITE on the per-tenant workspace directory (decided below)

WORKSPACE DECISION (planning call, per gap definition): The daemon assigns a per-tenant workspace under `%USERPROFILE%\nono-agents\<tenant_id>` — user-owned (R-B3), created by the daemon at launch time (before spawn), guaranteed unique per tenant. This is distinct from `%TEMP%` (acceptable but less predictable) and simpler than a caller-specified path (no `--workspace` flag on `agent launch`). The daemon creates this directory with `std::fs::create_dir_all`. Record this as a planning decision in the plan's task action.

CapabilitySet for daemon path: use `FsCapability::new_dir(path, AccessMode::Read)` for engine/interpreter/system dirs; use `FsCapability::new_dir(workspace_path, AccessMode::ReadWrite)` for the workspace. Then `Sandbox::windows_filesystem_policy(&caps)` gives the `WindowsFilesystemPolicy` for DACL grant application.

Note: The `Sandbox::windows_filesystem_policy` function is `#[cfg(target_os = "windows")]` and lives in the `nono` crate — fully accessible from `nono-agentd`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| package SID → filesystem DACL | The AppContainer package SID (S-1-15-2-*) is granted ACEs on specific paths only; all paths are user-owned and path-component-validated before grant |
| daemon → workspace creation | Daemon creates %USERPROFILE%\nono-agents\<token> owned by the running user (R-B3 — WRITE_OWNER) |
| tenant A grants ≠ tenant B grants | Each tenant gets a unique package SID (fresh AppContainer per tenant); DACL ACEs are per-SID; no cross-tenant bleed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-75-07-01 | Elevation of Privilege | over-broad DACL grant | mitigate | Grant ONLY: engine exe parent dir (Read+Traverse), interpreter dirs (Read+Traverse), %SystemRoot%+System32+SysWOW64 (Read+Traverse), per-tenant workspace (ReadWrite+Traverse). Never grant C:\ root, entire user profile, or any parent not owned by the user. Verify via `path_is_owned_by_current_user` gate inside `AppliedDaclGrantsGuard`. |
| T-75-07-02 | Information Disclosure | DACL leak after agent reap | mitigate | Grants are stored in AgentTenant fields (AppliedDaclGrantsGuard, read_traverse_paths Vec); AgentTenant::Drop calls revoke_sid_on_path for read paths and the guard Drops revoke write/traverse. On daemon crash before reap: stale ACEs remain until next daemon startup — ACCEPTED (documented, low-risk: the package SID is unique per session and cannot be reused without re-creating the AppContainer profile). |
| T-75-07-03 | Tampering | TOCTOU on granted paths | mitigate | Paths are canonicalized via std::fs::canonicalize before grant where possible; path component joins (Path::join) used throughout, never string-based starts_with; ownership check (`path_is_owned_by_current_user`) runs immediately before grant (inside AppliedDaclGrantsGuard). |
| T-75-07-04 | Elevation of Privilege | cross-tenant grant bleed | mitigate | Each agent gets a fresh per-agent AppContainer profile → fresh unique package SID (T-74-04-02 mitigation already in place). DACL ACEs are per-SID. Tenant A's package SID is on NO path granted to tenant B. Verified structurally: each `launch_agent` call generates a new tenant_id and calls `create_app_container_profile(&profile_name)` with a unique profile_name. |
| T-75-07-05 | Denial of Service | fail-secure on grant error | mitigate | If any grant step fails, the suspended process is TerminateProcess'd and Err returned (same pattern as job-assign failure at step 6). No partial-grant zombie is left alive. |
| T-75-07-SC | Tampering | npm/pip/cargo installs | accept | No new external crate dependencies; code reuses existing `nono::grant_sid_*`, `nono::revoke_sid_on_path`, `AppliedDaclGrantsGuard`, `AppliedAncestorTraverseGuard` already in the codebase and audited. |
