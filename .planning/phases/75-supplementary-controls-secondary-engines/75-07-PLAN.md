<objective>
Fix GAP-75-B (PRIORITY): daemon-launched agents are capability-less. `handle_launch` in control_loop.rs passes `CapabilitySet::new()` (empty) to `launch_agent`, and `launch_agent` never applies package-SID DACL grants. A confined agent therefore has zero filesystem access beyond the AppContainer default (System32), causing real engines to die on startup (`python312.dll not installed`; `claude.exe` / `node.exe` would fail identically).

Purpose: Wire the daemon launch path to match the per-invocation `nono run` path: build a real CapabilitySet from the named profile, construct a minimal `WindowsFilesystemPolicy` covering the engine runtime + system dirs + per-tenant workspace, apply package-SID DACL grants (write+traverse on workspace, read+traverse on read-only paths) BEFORE ResumeThread, store the grants in a self-contained local RAII guard in AgentTenant for revocation on reap.

This is the `launch_agent` ↔ `create_process_containment` dedup carry-forward from Phase 74. The local RAII guard approach (rather than importing exec_strategy_windows guards) preserves the module-independence invariant stated in launch.rs lines 27-31: nono-agentd does not declare exec_strategy_windows.

Output: Modified control_loop.rs (caps + workspace derivation), modified launch.rs (local DaemonDaclGuard struct + DACL grant application), modified reap.rs (dacl_guard field + Drop revocation). Three new unit tests for: caps non-empty for known profile; DaemonDaclGuard mid-loop failure reverts already-applied grants (fail-secure); reap revokes both write-grant and traverse-grant paths. One `autonomous: false` live claude.exe coverage proof task.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| package SID → filesystem DACL | The AppContainer package SID (S-1-15-2-*) is granted ACEs on specific paths only; all paths are user-owned and path-component-validated before grant |
| daemon → workspace creation | Daemon creates %USERPROFILE%\nono-agents\&lt;token&gt; owned by the running user (R-B3 — WRITE_OWNER) |
| tenant A grants ≠ tenant B grants | Each tenant gets a unique package SID (fresh AppContainer per tenant); DACL ACEs are per-SID; no cross-tenant bleed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-75-07-01 | Elevation of Privilege | over-broad DACL grant | mitigate | Grant ONLY: engine exe parent dir (Read+Traverse), interpreter dirs (Read+Traverse), %SystemRoot%+System32+SysWOW64 (Read+Traverse), per-tenant workspace (ReadWrite+Traverse). Never grant C:\ root, entire user profile, or any parent not owned by the user. Enforced via path_is_owned_by_current_user gate inside DaemonDaclGuard.apply before every grant. |
| T-75-07-02 | Information Disclosure | DACL leak after agent reap | mitigate | Grants are stored in DaemonDaclGuard (write_applied Vec + traverse_applied Vec) held by AgentTenant::dacl_guard. AgentTenant::drop causes DaemonDaclGuard::drop which calls revert_all LIFO. On daemon crash before reap: stale ACEs remain until next daemon startup — ACCEPTED (documented, low-risk: the package SID is unique per session and cannot be reused without re-creating the AppContainer profile). |
| T-75-07-03 | Tampering | TOCTOU on granted paths | mitigate | Paths are canonicalized via std::fs::canonicalize before grant where possible; path component joins (Path::join) used throughout, never string-based starts_with; ownership check (path_is_owned_by_current_user) runs immediately before each grant inside DaemonDaclGuard.apply. |
| T-75-07-04 | Elevation of Privilege | cross-tenant grant bleed | mitigate | Each agent gets a fresh per-agent AppContainer profile → fresh unique package SID (T-74-04-02 mitigation already in place). DACL ACEs are per-SID. Tenant A's package SID is on NO path granted to tenant B. Verified structurally: each launch_agent call generates a new tenant_id and calls create_app_container_profile with a unique profile_name. |
| T-75-07-05 | Denial of Service | fail-secure on grant error | mitigate | If DaemonDaclGuard::apply returns Err, the suspended process is TerminateProcess'd (same pattern as job-assign failure at step 6). No partial-grant zombie is left alive. |
| T-75-07-SC | Tampering | npm/pip/cargo installs | accept | No new external crate dependencies; code reuses existing nono::grant_sid_*, nono::revoke_sid_on_path, nono::path_is_owned_by_current_user already in the codebase and audited.
