<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| nono-ts confinedRun caller → nono.exe subprocess | confinedRun delegates all confinement to nono.exe run — the same trusted binary used by nono-py. The caller supplies exe/args/allow/profile; nono.exe applies the OS-enforced sandbox. |
| confine() re-exec guard (NONO_ALREADY_CONFINED) | The guard prevents the confined child from re-entering confine(). Exact "1" match enforced. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-75-04-01 | Denial of Service | confine() infinite re-exec loop | mitigate | NONO_ALREADY_CONFINED guard uses exact string equality (== Ok("1")); any other value (empty, "1extra", "true") does NOT match, so the call proceeds. This is identical to the nono-py T-72-02-04 carry-forward mitigation. Negative test: set NONO_ALREADY_CONFINED="1extra" → confine proceeds (guard does not fire). |
| T-75-04-02 | Spoofing | Caller sets NONO_ALREADY_CONFINED=1 to bypass confine() | accept | By design: if the caller is already inside a nono.exe-managed confined process, the guard is correct to skip re-exec. The soundness invariant is: confine() must be called BEFORE any privileged handle is opened (Shape B contract). The binding docs must state this ordering. |
| T-75-04-03 | Tampering | nono.exe not found on PATH; NONO_EXE points to malicious binary | mitigate | find_nono_exe() validates the NONO_EXE path with is_file() check; the operator is responsible for ensuring nono.exe on PATH is the trusted binary (same as nono-py). The R-B4 broker trust gate applies at the nono.exe level (it verifies its own binary signature). |
| T-75-04-SC | Tampering | npm/pip/cargo installs | mitigate | Only pin change: nono 0.33.0 → 0.62 (internal crate, not external registry). Package legitimacy audit confirms zero new external packages. |
</threat_model>
