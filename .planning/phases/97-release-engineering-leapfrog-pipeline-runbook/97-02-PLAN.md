Signing ORDER (Phase 53 UAT-B — MUST stay in this sequence):
  1. "Sign Windows binaries (pre-package) — Azure Trusted Signing"  signs target/<t>/release/*.exe in place
  2. "Package (Windows)"   build-windows-msi.ps1 harvests the now-signed binaries into the MSIs
  3. "Sign Windows MSIs — Azure Trusted Signing"  signs the two MSI wrappers
  4. "Verify Authenticode signatures (Windows)"   wrapper + loose exe must be Valid
  5. "Verify MSI payload signatures (Windows)"     msiexec /a admin-extract; every payload .exe must be Valid (the gate that was missing when the unsigned-payload MSI shipped)

0s startup_failure prevention (release.yml NOTE block ~lines 474-481): there is intentionally NO
docker job that does `uses: ./.github/workflows/image-build.yml`. image-build.yml has no
`on: workflow_call:` trigger, so a reusable call fails the WHOLE workflow at startup validation.
Do NOT re-add such a job.

Asset-name templates use the RELEASE_TAG env (e.g. nono-<tag>-x86_64-pc-windows-msvc-machine.msi) —
version-derived, so they track the tag automatically; verify they still resolve for a v0.66.0 tag.

crates.io publish-crates job (~lines 483-519) publishes ONLY nono, nono-proxy, nono-cli (in that
order, sleep 30 between). Publishable workspace crates are FOUR: nono, nono-proxy, nono-cli,
nono-shell-broker (none carry publish=false). Determine ANALYTICALLY whether nono-shell-broker must
join the publish set by reading crates/nono-cli/Cargo.toml: nono-shell-broker appears ONLY under
`[target.'cfg(target_os = "windows")'.dev-dependencies]` (version-pinned, line ~180) — i.e. it is a
DEV-dependency, not a normal/build dependency. cargo does NOT resolve dev-dependencies during a
crate's publish-verify build, and downstream consumers ignore dev-deps entirely — so a version-pinned
dev-dep does NOT force the depending crate into the publish set. Conclusion: the existing 3-crate set
(nono, nono-proxy, nono-cli) is sufficient for nono-cli to publish; nono-shell-broker need not be
added on nono-cli's account. This is decidable from the manifest alone — it does NOT require plan
97-03's dry-run output (97-03 runs in parallel in the same wave and merely corroborates).

| GitHub Actions startup validator → workflow graph | A malformed reusable-workflow call fails the whole release at 0s, silently shipping nothing. |
| homebrew formula → release tarball source repo | The download-url decides which GitHub repo's tarball brew fetches; a wrong/upstream repo serves an artifact the fork did not build. |

| T-97-05 | Spoofing | MSI wrapper vs payload signature | mitigate | Both wrapper (Verify Authenticode) and payload (admin-extract) verification steps retained; neither removed. |
