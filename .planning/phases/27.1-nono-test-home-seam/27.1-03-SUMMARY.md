## D-27.1-14 Contingency Triggered: Large-Fix Branch

D-27.1-14 specifies: "If localized fix is small (e.g., a session-discovery path that uses `dirs::home_dir()` inconsistently and was missed by the seam): fix in scope. If non-trivial: Document in plan SUMMARY, surface as v2.4 follow-up, mark the affected test with a temporary `#[ignore]` carrying a Phase 27.1-Blocker-3 note. **Do NOT block the seam landing on Blocker 3.**"

Two distinct downstream production-code gaps were surfaced — neither is a missed-callsite small fix, and both are documented as v2.4 follow-ups:

### Gap 1 (Test 1 failure): `audit verify` uses the wrong loader

**Symptom:** `cargo test ... audit_verify_reports_signed_attestation_with_pinned_public_key` succeeded at the `nono run --audit-integrity` step (the session was correctly created at `<NONO_TEST_HOME>/.nono/audit/<id>/audit-attestation.bundle`) but failed at the subsequent `nono audit verify <id> --json` invocation with `Session not found: <id>`.

**Root cause:** `crates/nono-cli/src/audit_commands.rs:12` imports `load_session` from `crate::rollback_session`, whose loader only inspects `<home>/.nono/rollbacks/<id>/`. Audit-only sessions (created by `--audit-integrity` without `--rollback`) live at `<home>/.nono/audit/<id>/` and are therefore unfindable.

**The audit-aware loader already exists:** `crates/nono-cli/src/audit_session.rs:160-198` (`audit_session::load_session`) implements correct dual-root semantics — it checks both audit and rollback roots. It is gated behind `#[allow(dead_code)]` and not yet wired into `audit_commands.rs`. Plan 22-05a's note at `audit_session.rs:16-18` acknowledged this: *"SessionInfo discovery is wired into audit_commands.rs by follow-up cherry-picks (4ec61c29..9db06336)"* — that wiring never landed.

**Why it's NOT a small fix:** Swapping the import in `audit_commands.rs` from `rollback_session` to `audit_session` is technically a one-line change, but it changes `audit list/cleanup/show/verify` semantics for ALL audit commands (they would discover audit-only sessions they previously missed). This requires:
- Removing 4 `#[allow(dead_code)]` attributes from `audit_session.rs`
- Verifying behavior is correct for `audit list/cleanup` (more sessions discoverable — generally correct, but deserves regression testing)
- Updating the existing audit_commands tests if any rely on the old single-root semantics

This is moderate-scope production work, not a missed-callsite fix. **Tracked as v2.4-FU-1.**

### Gap 2 (Test 2 failure): Bundle target mismatch for `--rollback --audit-sign-key` sessions

**Symptom:** `cargo test ... rollback_signed_session_verifies_from_audit_dir_bundle` failed at `only_audit_session_id(&home)` with `expected exactly one audit session: left: 0, right: 1`. The `nono run` step succeeded but the audit dir was empty.

**Root cause:** With `--rollback --audit-integrity --audit-sign-key`, `create_audit_state` (`rollback_runtime.rs:198-227`) routes the session to `<rollback>/<id>/` because `rollback_active = true` wins over `audit_integrity_requested`. `sign_session_attestation` (`audit_attestation.rs:155`) writes the bundle to `session_dir = <rollback>/<id>`. The bundle is in rollback dir; nothing exists in audit dir; the test's `only_audit_session_id` finds 0 sessions and panics.

**The test's design intent vs. production reality:** The test is named `rollback_signed_session_verifies_from_audit_dir_bundle`, suggesting the bundle SHOULD live in audit dir even for rollback sessions. But that's not what the code does. Either:
- (a) Mirror the bundle to audit_dir at sign time, OR
- (b) Make `audit verify` look up bundles in both roots, OR
- (c) Update the test to look in the rollback dir for the bundle.

**Why it's NOT a small fix:** Each option has architecture implications (does the audit-dir bundle survive rollback cleanup? Should `audit verify` check both roots even after v2.4-FU-1 lands? Does mirroring affect the chain-head signature contract?). This needs a design decision, not a quick fix. **Tracked as v2.4-FU-2.**

## Deviations from Plan

### Auto-applied Decisions

**1. [Decision] Pre-create `<NONO_TEST_HOME>/.nono/{audit,rollbacks}` in `setup_isolated_home`**

- **Issue:** Without these dirs pre-created, the supervisor's `--audit-integrity` exit-cleanup path attempted `canonicalize` on a non-existent rollback root, producing `Cannot canonicalize rollback root <path>: (os error 2)`.
- **Resolution:** Added `fs::create_dir_all(home.join(".nono").join("rollbacks"))` and `fs::create_dir_all(home.join(".nono").join("audit"))` to `setup_isolated_home` (D-27.1-14 Option A test-only fix).
- **Outcome:** Necessary but not sufficient — fixes the canonicalize symptom but reveals the deeper bugs (Gap 1, Gap 2 above). Kept the change in the committed file since it's defensive against future canonicalize-before-exists patterns.

**2. [Decision - D-27.1-14 large-fix branch] Re-#[ignore] BOTH tests, not just Test 1**

- **Issue:** The user's instructions assumed Test 2 would pass with just an `--audit-integrity` flag fix. Empirically, Test 2 ALSO fails because of Gap 2 (bundle in rollback_dir, test expects audit_dir). The user's hypothesis ("keep Test 2 working with the `--audit-integrity` fix") was incorrect — adding the flag fixes the CLI parse error but reveals the architectural gap underneath.
- **Resolution:** Re-#[ignore]'d both tests with separate Blocker-3-resurfaced notes pointing to v2.4-FU-1 (Test 1) and v2.4-FU-2 (Test 2). Test 2 keeps the `--audit-integrity` arg in its run_nono invocation since the CLI correctly requires it; the v2.4 re-enablement will need v2.4-FU-2's bundle-target decision.
- **Outcome:** Aligns with D-27.1-14's "Do NOT block the seam landing on Blocker 3" guidance. The seam itself is the deliverable; both production gaps are properly v2.4 work.
