<objective>
Close the SC 3 gap (CR-01 BLOCKER): path_covers() fails open when the child path (e.g., ~/.claude.json,
~/.claude.json.lock) does not exist on disk. On Windows, Path::canonicalize() fails for non-existent
paths; the current fallback returns the raw C:\... child while the parent is the \\?\-canonical CWD,
so Path::starts_with() always returns false — the guard silently reports "no risk" when it should deny.

This plan introduces canonicalize_with_existing_prefix() — a helper that walks up the child path until
it finds an existing ancestor, canonicalizes that ancestor, then re-appends the remaining tail
components. Both sides of the starts_with() comparison then use the same extended-length representation.

The plan also closes two adjacent fail-open paths (WR-01 and WR-02) in the same guard function that
share the same fail-secure theme: (1) missing home directory silently skips the entire home-state
limb; (2) a CWD nested inside ~/.claude is not caught by the one-directional ancestor check.

Purpose: Restores the SC 3 confinement invariant. D-05 (per CONTEXT.md) prohibits relying on
profile add_deny_access to carve out ~/.claude — the hook-level CWD guard is the sole mitigation.
Without this fix, the guard fails on the most common case (.claude.json.lock absent while no session
holds the lock), allowing an --allow-cwd grant that covers the home directory to re-introduce a
write path to ~/.claude hook state.

Output: Updated crates/nono-cli/src/claude_code_hook.rs with a new #[cfg(target_os = "windows")]
helper canonicalize_with_existing_prefix(), a patched path_covers(), two guard fixes in
cwd_covers_home_claude_state() and cwd_self_disable_risk_reason_for(), and two new regression tests.
</objective>

  CHANGE 1 — Add canonicalize_with_existing_prefix() helper (new #[cfg(target_os = "windows")] function,
  place immediately after canonical_existing_path()):

  The function signature is:
    fn canonicalize_with_existing_prefix(path: &Path) -> PathBuf

  Semantics: walk up the path's ancestor chain until a prefix that exists on disk is found, canonicalize
  that prefix via Path::canonicalize() (NOT via canonical_existing_path() — no error propagation needed
  here; if even the filesystem root fails to canonicalize, return the path as-is so the guard stays
  conservative), then re-join the remaining tail components. Both the existing-ancestor canonicalization
  and the tail re-join must use Path component operations, not string manipulation.

  Algorithm:
    1. Let ancestors = path.ancestors() — iterate from path itself down to the root.
    2. For each ancestor in ancestors:
       a. If ancestor.exists(), call ancestor.canonicalize(). If it succeeds, collect the remaining
          suffix components (path.components() minus the ancestor.components() count) and join them
          onto the canonical_base. Return that joined path.
       b. If canonicalize fails, continue to the next ancestor (degrade gracefully toward root).
    3. If no ancestor canonicalizes (degenerate case — isolated in-memory path), return path.to_path_buf().
       This is conservative: the caller (path_covers) will then compare two paths that may differ in
       form, which Path::starts_with() may still catch if they happen to share the same raw prefix.
       Worst case the guard passes (does not fire), but this only happens if even the filesystem root
       is non-canonicalizable — an extraordinary condition.

  NOTE ON SAFETY: Do NOT use .unwrap() or .expect(). Use .ok() to discard errors from canonicalize()
  inside the loop. The function returns PathBuf (not Result) — all error paths fall through to the
  conservative return of path.to_path_buf().

  CHANGE 2 — Replace the body of path_covers():

  Current body (the bug):
    let child = canonical_existing_path(child).unwrap_or_else(|_| child.to_path_buf());
