<objective>
Implement confined file-operation arms for Write, Edit, and MultiEdit in the PreToolUse hook handler.
Currently these tools are caught by the blanket deny-all guard at line 56. This plan changes that guard
into a dispatch structure: Write/Edit/MultiEdit emit deny + additionalContext (a PowerShell nono-run
command Claude can retry as a Bash call); NotebookEdit emits an informative deny (not yet supported,
no PS command). The D-05 CWD self-disable guard fires first in every new arm, before any command
is constructed.

Purpose: Turns denied file edits into confined Low-IL file operations, gating the phase's core deliverable
(REQ-STW-01). Once this lands, Claude can complete a write-then-read coding loop without manual
intervention from the user.

Output: Updated claude_code_hook.rs with three new Windows-gated helper functions
(build_confined_write_cmd, build_confined_edit_cmd, build_confined_multiedit_cmd), a dispatch
restructure replacing the `if tool_name != "Bash"` guard, new unit tests in the existing mod tests block,
and an updated pre_tool_use_file_tools_deny assertion that verifies additionalContext for Write/Edit/MultiEdit.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Claude TUI (Medium-IL) → hook handler | tool_name and tool_input arrive as untrusted JSON from the agent's context; a prompt-injected instruction could craft arbitrary tool_input values |
| hook handler → PS command string | The hook constructs a PowerShell command string that is base64-encoded and passed to nono run; file_path and content from tool_input are attacker-controllable |
| nono run → Windows MIC kernel | The CWD grant is the OS enforcement boundary; paths outside the CWD grant are denied by the mandatory-label layer, not by the hook |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-60-01 | Tampering | Write arm — file_path value from tool_input | mitigate | file_path is passed through powershell_single_quoted() which escapes embedded single quotes; the Windows MIC CWD grant is the OS-level boundary for path escape attempts. A path pointing outside the CWD is denied by the kernel, not by the hook. No additional string-starts_with check is needed (and would be incorrect per CLAUDE.md path-comparison rule). |
| T-60-02 | Tampering | Write arm — content value from tool_input | mitigate | Content is base64-encoded before embedding in the PS command (build_confined_write_cmd uses base64_encode + [Convert]::FromBase64String decode in the PS command). Embedded single quotes, newlines, and PS metacharacters cannot escape the encoding layer. |
| T-60-03 | Tampering | Edit arm — old_string/new_string values from tool_input | mitigate | old_string and new_string are passed through powershell_single_quoted(); the PS command uses String .Replace() (literal, not -replace regex) to eliminate regex injection (RESEARCH.md Pitfall 1). |
| T-60-04 | Tampering | Self-disable via Write to ~/.claude hook state | mitigate | cwd_self_disable_risk_reason() is the FIRST call in each confined arm (D-05). On Windows, if the process CWD covers ~/.claude or the project .claude/ child, the arm returns deny_response(reason) WITHOUT building any PS command. The guard is identical to the existing Bash arm guard and is covered by 3 of the 8 existing tests. |
| T-60-05 | Tampering | Path escape via ../.. in file_path | mitigate | The Windows MIC CWD grant is the enforcement boundary (not a hook-level string check). An attempt to write to ../../etc/passwd results in a Low-IL mandatory-label denial at the OS boundary — the correct fail-closed outcome. The hook does not need to validate the path beyond extracting it safely. |
| T-60-06 | Elevation of Privilege | NotebookEdit — complex JSON manipulation that could become a vector | accept | NotebookEdit is kept as an informative deny with no PS command in this phase (RESEARCH.md Open Question 3). No confinement path is wired; the tool stays denied. Risk: none for this phase. |
| T-60-07 | Tampering | MultiEdit partial-edit state corruption | mitigate | build_confined_multiedit_cmd applies ALL edits in-memory to $content and writes once via [System.IO.File]::WriteAllText (RESEARCH.md Pitfall 4). No interleaved read/write; the file is in a consistent state after the single write or unchanged if the PS command fails. |
| T-60-SC | Tampering | npm/pip/cargo installs | accept | No new external packages in this phase (RESEARCH.md: "No new external packages"). No package-manager install tasks. |
