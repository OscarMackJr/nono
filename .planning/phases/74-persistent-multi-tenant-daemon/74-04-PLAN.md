The file implements three public(crate) functions:

FUNCTION 1: `create_daemon_capability_pipe_instance(base_sddl: &str) -> Result<HANDLE>`
Creates a new `CreateNamedPipeW` instance using the base Low-IL SDDL (not per-tenant — we don't know the tenant yet at connection time). Use `PIPE_UNLIMITED_INSTANCES`, `FILE_FLAG_OVERLAPPED`, `MAX_MESSAGE_SIZE` (same constant as socket_windows.rs). Apply the `SECURITY_ATTRIBUTES` from the SDDL via the existing `SecurityDescriptorHandle` + `SECURITY_ATTRIBUTES` construction pattern in `socket_windows.rs` (grep for `ConvertStringSecurityDescriptorToSecurityDescriptorW` in socket_windows.rs to find the pattern).

Pipe name: `\\.\pipe\nono-agentd-cap-<nonce>` where nonce is a random 8-char hex string (use `getrandom` or `rand` — already in the workspace).

FUNCTION 2: `run_accept_loop(daemon_state: Arc<DaemonState>, shutdown: Arc<tokio::sync::Notify>)`
Async fn. Loop:
1. Call `create_daemon_capability_pipe_instance(base_sddl)` — create a new pipe instance.
2. Wrap in `tokio::net::windows::named_pipe::NamedPipeServer::from_raw_handle` or use `ServerOptions::new()` for the accept.
3. `tokio::select! { biased; _ = shutdown.notified() => break; _ = server.connect() => {} }` — interruptible by shutdown signal.
4. Clone `Arc<DaemonState>`; `tokio::spawn(async move { handle_one_connection(pipe_handle, state).await })`.
5. Loop back to create the next instance.

FUNCTION 3: `handle_one_connection(pipe_handle: HANDLE, state: Arc<DaemonState>) -> Result<()>`
1. Call `nono::supervisor::socket_windows::authenticate_pipe_client(pipe_handle)` → `client_sid: String`.
2. On Err: log `tracing::warn!("Pipe auth failed: {e}"); close pipe; return Ok(())` (fail-secure: deny silently).
3. Look up tenant by SID: `state.tenants.lock()?.values().find(|t| t.package_sid == client_sid).cloned()`.
4. On None: `tracing::warn!("SID not in tenant registry: {client_sid}"); close pipe; return Ok(())`.
5. On Some(tenant): serve capability frames using the existing framed-JSON `SupervisorMessage` reading pattern from `socket_windows.rs`. The `session_id` from the wire frame is used ONLY as a routing hint to look up the tenant entry — authorization is already done by SID match in step 3-4. The response capability set is read from `tenant.caps`.
6. On any frame-serving error: log warn + close pipe + return Ok(()).

All frame reading should use the existing `read_framed`/`write_framed` functions in `socket_windows.rs` if they are pub(crate). If not, adapt the pattern.

The base SDDL used for the initial pipe instance (before tenant SID is known) should admit the daemon's own session + Low-IL same-session processes. After authenticating the tenant SID, the DAEMON verifies the SID is in the registry — this is defense-in-depth on top of the accept. (The per-tenant SDDL ACE approach is also valid — see RESEARCH.md alternatives — Claude's discretion.)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Wire-frame session_id → authz decision | session_id is untrusted; authz is always by kernel-vouched SID from authenticate_pipe_client |
| accept_loop → tenant registry | SID not in registry → deny; fail-secure default for all unknown connections |
| launch_agent → broker trust gate | A6 resolution determines call path; fail-secure on assign failure (terminate the suspended process) |
| reap_task → AgentTenant::Drop | WaitForSingleObject signals → tenant removed → Drop → KILL_ON_JOB_CLOSE + DeleteAppContainerProfile |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-74-04-01 | Spoofing | Wire-frame session_id used as authz signal in accept loop | mitigate | Test `session_id_is_routing_hint_not_authz`; `handle_one_connection` always authorizes by SID from `authenticate_pipe_client`, never by `session_id` field value |
| T-74-04-02 | Elevation of Privilege | AppContainer profile reuse across agents (token reuse) | mitigate | Each `launch_agent` call generates a new session_id (unique nonce) → new profile name → distinct AppContainer SID; test `launch_agent_fresh_profile_per_agent` asserts distinct SIDs |
| T-74-04-03 | Denial of Service | Agent process not terminated on daemon reap task failure | mitigate | `KILL_ON_JOB_CLOSE` on the job handle ensures termination even if the reap task panics — AgentTenant::Drop closes job_handle regardless |
| T-74-04-04 | Elevation of Privilege | Suspended process not terminated on job-assign failure | mitigate | `apply_process_handle_to_containment` fail-secure behavior (from exec_strategy_windows/launch.rs): terminates the suspended process on assign failure; daemon inherits this behavior |
| T-74-04-05 | Information Disclosure | Capability grants served to wrong tenant (SID match bypassed) | mitigate | `handle_one_connection` finds tenant by SID (kernel-vouched), not by session_id; test `accept_loop_denies_unknown_sid` verifies unknown SID → deny |
| T-74-SC | Tampering | cargo installs | accept | No new packages |
