Create proj/ADR-74-privilege-model.md as a free-form ADR document (no rigid template; follow the narrative style of DESIGN-engine-abstraction.md). The ADR must record:

Status: Accepted

Context: The `nono-wfp-service` runs as LocalSystem (SYSTEM) because it needs `FwpmFilterAdd0` WFP kernel-driver access. The agent daemon (nono-agentd) has NO WFP responsibility (D-04: per-agent WFP egress is Phase 75). Running the daemon as SYSTEM would be Pitfall 4: an escaped Low-IL agent that reaches the daemon could pivot to SYSTEM and to all other tenants.

Decision 1 — USER privilege, not SYSTEM: The daemon registers as a per-user Windows service (`SERVICE_USER_OWN_PROCESS`) running under the interactive user's session logon token. It never elevates. Per D-01 (locked): the USER service is fully split from the elevated `nono-wfp-service` (which remains the only SYSTEM-privilege service in the MSI).

Decision 2 — Foreground fallback: When `service_dispatcher` fails (binary not running in SCM context), the daemon falls through to foreground/on-demand mode without panicking. This is the dev/testing path. Mirrors the non-fatal posture in `nono-wfp-service.rs`.

Decision 3 — Query-only pipe, no escape hatch: The capability pipe accepts capability QUERY requests only. It never expands a running agent's capabilities in response to a wire request. Once an agent's `AgentTenant` is inserted, its `CapabilitySet` is immutable. A request to ADD capabilities to a running agent is denied (wire-frame authz boundary). This is the "no escape hatch" invariant from SC4.

Decision 4 — Split from WFP service: The USER daemon binary (`nono-agentd`) MUST NOT import or call any WFP Win32 APIs (`FwpmFilterAdd0`, `FwpmEngineOpen0`, etc.) in Phase 74. Network policy in Phase 74 is profile-only (D-04). Any daemon→WFP coupling is Phase 75 / SUPP-02.

Decision 5 — `SeImpersonatePrivilege`: The daemon's `ImpersonateNamedPipeClient` call requires `SeImpersonatePrivilege`. Interactive users and their services have this privilege by default on Win10/11. Assumption A1 (per 74-RESEARCH.md Open Questions) must be confirmed during the Wave 0 spike. If the privilege is absent in the per-user service token, the workaround is SDDL-only + `GetNamedPipeClientProcessId` + `IsProcessInJob` (no impersonation; per-tenant SDDL remains the primary gate regardless).

Consequences: The daemon binary is named `nono-agentd`; it is a second `[[bin]]` target in `crates/nono-cli/Cargo.toml`. It does NOT appear in the machine-scoped `nono-machine.wxs` MSI alongside `nono-wfp-service`. Installation is via `nono daemon install` CLI verb (Phase 74) or a per-user MSI (Phase 75). An escaped agent reaching the daemon process gets USER-level access, not SYSTEM — it cannot pivot to other sessions or WFP filter rules.
