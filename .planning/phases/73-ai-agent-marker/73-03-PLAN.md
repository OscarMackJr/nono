    ## AI_AGENT Marker: Scope and Adopted-Agent Limits

    The `AI_AGENT` marker is sound **only for agents the launcher spawned**. An agent is marked by inserting its per-run AppContainer package SID (minted at `BrokerLaunchNoPty` spawn time) into the launcher's in-memory `AgentRegistry`. The registry is the authorization predicate.

    ### Standalone `nono classify` is structural/non-authoritative

    A standalone `nono classify <pid>` invocation runs in a **separate process** from the launcher. Its `AgentRegistry` is always empty (created fresh, no inserts). The command therefore **cannot** make a registry-backed `AI_AGENT` authorization claim cross-process. It performs a structural pre-filter only:
    - Is the PID's token an AppContainer token?
    - Is the process in a job object?

    A "structural match" means both conditions hold. This is **non-authoritative**: a self-created AppContainer in a self-created job passes the structural filter but is NOT in any launcher's `AgentRegistry`. The output always includes a disclaimer: "(NOTE: This check is structural only -- not an authorization decision. Registry-backed authoritative classification requires Phase 74 daemon.)"

    Registry-backed cross-process authoritative classification is explicitly Phase 74 (daemon scope -- the persistent daemon shares its `AgentRegistry` across clients).

    ### Adopted agents (not launched by this nono instance)

    An "adopted" agent is a process nono did not spawn -- for example, a pre-existing running AI agent that an operator wants to retroactively confine or classify.

    **Current behavior (best-effort only):**
    - `nono classify <pid>` performs a structural pre-filter: it reads the PID's AppContainer package SID and checks whether the process is in a job object.
    - The structural pre-filter is **non-authoritative**: a self-created AppContainer passes the structural filter but is NOT in the `AgentRegistry` and therefore cannot be authoritatively classified as `AI_AGENT`.
    - There is no sound way to adopt a running agent and retroactively grant it `AI_AGENT` status -- doing so would require adding its SID to the registry after the fact, which creates a forgery window.

    **Demote-only model (per spike-002 findings):**
    Post-hoc token IL-drop (demote) is an incident-response lever, not a confinement model. Demote is a one-way operation: reducing IL does not retroactively confine all prior operations. Adopted-agent demote is implemented in Phase 75 (SUPP-01) as a supplementary control layered on a proven spawn-time default.

    **Recommendation:** To confine a new agent session, always launch it through `nono run` (or the `nono-py`/`nono-ts` bindings). Do not rely on post-hoc adoption as a primary security boundary.
