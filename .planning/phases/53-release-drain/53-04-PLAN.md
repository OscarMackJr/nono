<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Release artifacts → end-user download | Artifacts must be Authenticode-signed; unsigned MSIs must never be uploaded (D-13) |
| Installed nono.exe → broker self-trust-anchor | verify_broker_authenticode (D-32-12): installed binary + broker must both be signed by the same cert chaining to LocalMachine\Root + TrustedPublisher |
| Elevated WFP uninstall → kernel state | The WFP service/driver removal runs at kernel-level; fail-open (Return="ignore") is correct behavior — a CA failure degrades to "services left behind", not "broken uninstall" |
| Fresh cert redistribution → broker gate | If the .cer is not imported into LocalMachine\Root + LocalMachine\TrustedPublisher on the target machine, WinVerifyTrust will reject the signed binaries and the broker gate will fail-close (expected, correct) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-53-04-01 | Information Disclosure | PFX private key in CI run logs | mitigate | GitHub Actions masks the WINDOWS_SIGNING_CERT secret; sign-windows-artifacts.ps1 deletes temp PFX in finally block; operator must verify no secret value appears in CI log output |
| T-53-04-02 | Tampering | Unsigned artifacts uploaded if signing secrets missing | mitigate | D-13 gate preserved in release.yml — "Check signing secrets" exits 1 on missing secrets, blocking all downstream steps including artifact upload |
| T-53-04-03 | Elevation of Privilege | WFP driver removal grants elevated access | accept | WFP custom action runs deferred (no impersonation), fail-open (Return="ignore"); removal is the intended outcome; fail-open means broken CA degrades to "driver left behind" (today's behavior), not elevated-privilege escape |
| T-53-04-04 | Spoofing | Duplicate CI run for v2.8 milestone tag | mitigate | release.yml trigger refined to v*.*.* (Plan 53-03); UAT-A verifies only one workflow run fires |
| T-53-04-05 | Denial | Stale Cargo.lock causes build failure on CI | accept | Cargo.lock regenerates from bumped Cargo.toml on first `cargo build`; CI runs cargo build from checkout of the tag ref which has the updated lock |
| T-53-04-SC | Tampering | npm/pip/cargo installs | accept | No new package installs in this plan; human verification only |
</threat_model>
