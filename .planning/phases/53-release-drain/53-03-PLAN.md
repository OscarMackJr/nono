    The fork is wired for a production commercial CA swap that requires only a secrets change,
    no code change. To upgrade from the POC self-signed cert to a commercial code-signing
    cert (e.g. DigiCert, Sectigo, or Azure Trusted Signing):
    1. Obtain a code-signing certificate from a CA enrolled in the Microsoft Trusted Root
       Program (or configure Azure Trusted Signing with a verified publisher account).
    2. Export the certificate as a PFX file (certificate + private key, password-protected).
    3. Base64-encode the PFX: run
       `[Convert]::ToBase64String([IO.File]::ReadAllBytes("cert.pfx"))`
    4. Update the `WINDOWS_SIGNING_CERT` repository secret with the new base64 value.
    5. Update `WINDOWS_SIGNING_CERT_PASSWORD` with the new PFX password.
    6. No code changes are required. The release workflow (`release.yml`) reads these two
       secrets and passes them directly to `scripts/sign-windows-artifacts.ps1` — the signing
       path is identical whether the cert is self-signed or CA-issued.
    7. For Azure Trusted Signing: consult the Azure Trusted Signing documentation for the
       `signtool.exe`-compatible dlib invocation; the `sign-windows-artifacts.ps1` script may
       need a small extension to pass the `-dlib` and `-dmdf` flags instead of `/sha1`.
       Document that extension here when it is implemented.

    Section 2 — "Fresh CI cert: generation, encoding, and trust redistribution" (D-53-02):

    Add a section titled `## Fresh CI cert: generation, encoding, and trust redistribution`
    with this content:

    D-53-02: each v2.x release uses a fresh self-signed cert (provenance born in/for CI),
    NOT a re-export of a previous POC cert. Consequence: the new cert's public `.cer` MUST
    be redistributed to every trusted machine before they accept the new signed binaries.
    Old installs' trust does not carry over automatically.

    Step 1: Generate the fresh cert (elevated PowerShell on the operator's machine):

    Use the cert-gen block from `sign-poc-local.ps1` (lines 137-144): run
    `New-SelfSignedCertificate` with subject `CN=nono POC Signing v0.57.4` (or similar
    version-tagged subject to distinguish certs across releases), type CodeSigningCert,
    HashAlgorithm SHA256, NotAfter two years from now. The cert lands in
    Cert:\CurrentUser\My.

    Step 2: Export the PFX (private + public key, password-protected):

    Run `Export-PfxCertificate -Cert $cert -FilePath nono-v0.57.4-ci.pfx -Password $password`
    where `$password = ConvertTo-SecureString "YourPassword" -AsPlainText -Force`.

    Step 3: Base64-encode the PFX for the secret:

    Run `[Convert]::ToBase64String([IO.File]::ReadAllBytes("nono-v0.57.4-ci.pfx"))` and copy
    the output. This is the value for the `WINDOWS_SIGNING_CERT` repository secret.

    Step 4: Set GitHub repository secrets:

    Navigate to Settings -> Secrets and variables -> Actions. Create or update:
    - `WINDOWS_SIGNING_CERT` — the base64 PFX value from Step 3.
    - `WINDOWS_SIGNING_CERT_PASSWORD` — the plaintext password from Step 2.
    The PFX file must be securely deleted after encoding: `Remove-Item nono-v0.57.4-ci.pfx`.

    Step 5: Export the public `.cer` for trust redistribution (NO private key):

    Run `Export-Certificate -Cert $cert -FilePath nono-v0.57.4-poc.cer -Type CERT`.
    Distribute this `.cer` to every machine that needs to accept v0.57.4-signed binaries.
    On each target machine (admin, one-time):
      `Import-Certificate -FilePath nono-v0.57.4-poc.cer -CertStoreLocation Cert:\LocalMachine\Root`
      `Import-Certificate -FilePath nono-v0.57.4-poc.cer -CertStoreLocation Cert:\LocalMachine\TrustedPublisher`
    After import, `(Get-AuthenticodeSignature 'C:\Program Files\nono\nono.exe').Status` must
    report `Valid`. Without this step, the broker self-trust-anchor gate (D-32-12) will
    fail-close on the new binaries even on machines that accepted previous signed releases.

    Step 6: Push main, then cut the dual tags:

    Local `main` must be pushed before cutting the semver tag (the release workflow checks

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer machine → GitHub Secrets | PFX private key must never appear in logs, artifacts, or git history; only the base64 PFX goes into the secret |
| release.yml signing gate → artifact upload | D-13: fail-closed gate must not be weakened; any change that bypasses "Check signing secrets" would allow unsigned artifact upload |
| Cert trust store → broker trust anchor | D-32-12: the fresh cert's public .cer must be redistributed before v2.8-signed binaries pass verify_broker_authenticode |
| git push origin main → tag firing CI | The tag ref must be reachable from origin for release.yml to check it out successfully |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-53-03-01 | Information Disclosure | WINDOWS_SIGNING_CERT PFX exposure in logs | mitigate | Secret is masked by GitHub Actions; release.yml never echoes the cert value; sign-windows-artifacts.ps1 writes to temp file and deletes in finally block |
| T-53-03-02 | Tampering | release.yml signing gate weakened by trigger change | mitigate | Only the tags glob is changed (v* → v*.*.*); accept-criteria explicitly require the signing gate (lines 124-139) is unchanged |
| T-53-03-03 | Elevation of Privilege | Unsigned artifact upload if signing gate bypassed | mitigate | D-13 gate preserved; the "Check signing secrets" step exits 1 if either secret is missing or blank; no upload step precedes signing verification |
| T-53-03-04 | Tampering | v2.8 milestone tag accidentally fires CI build | mitigate | v*.*.*  glob does not match v2.8 (two-segment); verify step explicitly confirms bare v* is gone |
| T-53-03-05 | Spoofing | Stale cert on trusted machines accepts old-signed binaries | accept | Old cert still valid for previously-installed binaries; redistribution of new .cer extends trust to new binaries; machines can remove old cert if desired |
| T-53-03-SC | Tampering | npm/pip/cargo installs | accept | No package installs; YAML + script + MDX text edits only |
</threat_model>
