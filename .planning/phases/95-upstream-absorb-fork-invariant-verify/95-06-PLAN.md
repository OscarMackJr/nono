<objective>
Fix two fork-invariant violations introduced by the Cluster A/B cherry-pick: the CR-01
fork-safety violation where format!() heap allocation was introduced in the post-fork/pre-exec
child (GAP: WR-01), and the operator fail-open where CompiledEndpointPolicy.evaluate() has
zero callers despite being a live operator-configurable security control (GAP: CR-01).

Purpose:
  WR-01: The post-fork/pre-exec child MUST NOT heap-allocate. Adjacent code (lines 1064-1139)
  has explicit CR-01 comments establishing this invariant with const MSG_* static byte strings.
  Two new error paths (lines 1411-1414 and 1450-1451) introduced by the cherry-pick use format!()
  instead, which can deadlock if the allocator mutex was held at fork() time. This is an
  async-signal-safety violation.

  CR-01: The CompiledEndpointPolicy struct has an evaluate() method compiled at startup and
  stored in LoadedRoute.endpoint_policy. However reverse.rs:96 calls the legacy
  route.endpoint_rules.is_allowed() and never calls route.endpoint_policy.evaluate().
  An operator who configures endpoint_policy.deny rules gets them silently ignored. This is
  a fail-open in the security proxy. Option (a) — wire evaluate() into the request path —
  is the correct choice: endpoint_policy is already compiled at startup, stored in LoadedRoute,
  and exposes a complete deny/approve/allow/default precedence model. Fail-closed option (b)
  would be appropriate only if endpoint_policy were premature surface; here it is a complete,
  compiled, stored feature that simply needs to be called.

Output: exec_strategy.rs with const static byte strings at both post-fork error paths;
reverse.rs with endpoint_policy.evaluate() called after the existing endpoint_rules check
(additive: evaluate() enforces the new policy, endpoint_rules preserves legacy behavior);
a test in nono-proxy proving a configured deny rule is actually enforced.
</objective>
