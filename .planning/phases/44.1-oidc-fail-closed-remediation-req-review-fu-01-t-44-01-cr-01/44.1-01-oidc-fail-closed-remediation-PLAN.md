## Shape A vs Shape B — DECISION: Shape A++ (helper extraction)

**Selected:** Shape A (inline env-var read at both verify callsites in `trust_cmd.rs`),
with a small refactor — extract the env-var read into a private helper
`fn read_required_oidc_issuer(user_issuer: Option<&str>) -> std::result::Result<String, String>`
in `crates/nono-cli/src/trust_cmd.rs`. Both keyless verify callsites call this
helper. The helper is unit-testable directly in `trust_cmd.rs`'s `#[cfg(test)] mod tests`.

**Why not Shape B (split `configured_oidc_issuer` into `_or_default()` + `_required()`):**

**Only two production call sites exist; BOTH are verify-path call sites.** There
are zero non-verify callers that "need a default". Introducing Shape B's
`_or_default()` variant would create dead code in the library (CLAUDE.md
§ Coding Standards → "Lazy use of dead code: Avoid `#[allow(dead_code)]`. If
code is unused, either remove it or write tests that use it."). The bare
`_required()` variant returning `Option<String>` would simply be the env-var
read with `filter(|v| !v.trim().is_empty())` and `url::Url::parse` validation —
which is exactly what the helper in Shape A++ does, but lives in `trust_cmd.rs`
(CLI) rather than `signing.rs` (library). That placement is correct per the
CLAUDE.md library/CLI boundary: "Library is policy-free; verify-path decisions
live in the CLI" (Key Design Decision #4). Shape A++ keeps the policy on the
right side of the boundary.

**Why also delete `configured_oidc_issuer()` entirely:**

After Shape A++ extracts the helper, the existing `configured_oidc_issuer()`
function has zero callers in production code. Its 4 tests in signing.rs are the
only consumers. CLAUDE.md "lazy dead code" forbids `#[allow(dead_code)]` on a
public API with no callers; the function must either be removed or repurposed.
Repurposing it (e.g., to "return `Option<String>` without a default") makes it
functionally identical to the inline helper in `trust_cmd.rs` and only adds a
library/CLI hop with no benefit. Decision: **delete the function and all 4 of
its tests in signing.rs**. The malformed-env coverage migrates to the helper's
test in `trust_cmd.rs` (which inlines the same `url::Url::parse` logic).

This also satisfies the security audit's CLAUDE.md § "Library is policy-free"
posture: removing a library function whose only behavior was to silently
substitute a hard-coded trust anchor is the cleanest answer to T-44-01.

**Compatibility check:** `configured_oidc_issuer` is `pub` in the library, but
the only consumers are inside this workspace (verified by grep above). There
are no language-binding consumers (PyO3 `nono-py` / napi-rs `nono-ts` bindings
expose only sandbox-policy / capability primitives, not trust internals).
Removing it is a non-breaking change for downstream consumers.

**`GITHUB_ACTIONS_OIDC_ISSUER` constant:** stays as `pub const`. It is still a
useful pin reference for tests and for the updated CLI doc string ("Example:
https://token.actions.githubusercontent.com"). Its removal is not a security
requirement; only its silent use as a default at the verify boundary is.

        use super::*;

        // Phase 44.1 — env-var test isolation. Replicates the OidcEnvGuard

        static OIDC_ENV_LOCK: std::sync::Mutex<()> = std::sync::Mutex::new(());

        struct OidcEnvGuard {
            _lock: std::sync::MutexGuard<'static, ()>,
            prev: Option<String>,
        }

        impl OidcEnvGuard {
            #[allow(clippy::disallowed_methods)]
            fn set(val: &str) -> Self {
                let _lock = match OIDC_ENV_LOCK.lock() {
                    Ok(g) => g,
                    Err(p) => p.into_inner(),
                };
                let prev = std::env::var("NONO_TRUST_OIDC_ISSUER").ok();
                std::env::set_var("NONO_TRUST_OIDC_ISSUER", val);
                Self { _lock, prev }
            }

            #[allow(clippy::disallowed_methods)]
            fn remove() -> Self {
                let _lock = match OIDC_ENV_LOCK.lock() {
                    Ok(g) => g,
                    Err(p) => p.into_inner(),
                };
                let prev = std::env::var("NONO_TRUST_OIDC_ISSUER").ok();
                std::env::remove_var("NONO_TRUST_OIDC_ISSUER");
                Self { _lock, prev }
            }
        }

            fn drop(&mut self) {

                    None => std::env::remove_var("NONO_TRUST_OIDC_ISSUER"),

    ///
    /// CLAUDE.md § Fail Secure + § Explicit Over Implicit: the keyless verify
    /// path MUST NOT silently substitute a hard-coded canonical OIDC issuer
    /// (e.g. `https://token.actions.githubusercontent.com`) when both inputs
    /// are absent. Such a substitution would allow an operator who forgot the
    /// `--issuer` flag to succeed against any GitHub-Actions-keyless bundle
    /// they did not explicitly choose to trust. The pre-Phase-44 behavior

    ///

    ///   `url::Url::parse`.
    fn read_required_oidc_issuer(
        user_issuer: Option<&str>,

        if let Some(s) = user_issuer {
            return Ok(s.to_string());

            .filter(|v| !v.trim().is_empty())

                "keyless bundle requires --issuer <OIDC_URL> or NONO_TRUST_OIDC_ISSUER \
                 env-var (exact match against signer's iss claim)"

        // letting `validate_oidc_issuer` produce a less specific message
        // downstream. Mirrors the historical

            format!("NONO_TRUST_OIDC_ISSUER='{env_value}' is not a valid URL: {e}")
        })?;
        Ok(env_value)
    }
