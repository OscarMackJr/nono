<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Rust verifier Result → Python caller | Every `Err` must surface as a raised exception, never a falsy/None return that a caller could treat as "allow" |
| OverrideGrant → Python | The grant must be immutable from Python so a caller cannot widen scope after verification |
| exception message → logs | The raised message must not leak raw secrets/key bytes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-91-03-FALSY | Elevation of Privilege | PyO3 boundary return | mitigate | verify_override `#[pyfunction]` returns `PyResult`; every `Err` → raised `NonoOverrideError` via `override_err_to_py`; no path returns None/False/Ok-with-deny (PITFALLS #1 / SC4). |
| T-91-03-WRONGEXC | Spoofing | exception type | mitigate | A single dedicated `NonoOverrideError` (not a built-in) is raised for every variant; callers can reliably catch the override-specific type; args[0] = stable kind for Phase 92's EventID-10008 map (D-04). |
| T-91-03-MUTGRANT | Tampering | OverrideGrant immutability | mitigate | `#[pyclass(frozen)]` — Python cannot mutate the grant to widen scope after verification (D-02 / PITFALLS #6). |
| T-91-03-LEAK | Information Disclosure | exception message | mitigate | `override_err_to_py` message is redaction-safe (no raw signature/key bytes; paths per existing redaction policy); `__repr__` likewise. |
| T-91-03-MUSTUSE | Elevation of Privilege | ignored Result | mitigate | `#[must_use]` on verify_override (Plan 02) + an SC5 negative-control proving `unused_must_use` fires — a caller cannot silently drop a deny. |
</threat_model>
