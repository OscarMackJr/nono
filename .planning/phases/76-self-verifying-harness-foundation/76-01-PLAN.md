    Create `scripts/verify-dark.ps1` as a thin runner. Header: `$ErrorActionPreference = "Stop"`
    plus `$PSNativeCommandUseErrorActionPreference = $false` (per windows-test-harness.ps1:7-10) so
    a future gate shelling out does not promote benign native stderr to a terminating error. Param
    block: `param([string]$Gate, [switch]$All)` — a plain `[string]$Gate` (NOT a hardcoded
    `ValidateSet`, because gates are auto-discovered per D-04); treat "no `-Gate`" as the all-run
    (Claude's discretion). For this plan the all-run path may be a minimal stub that loops the
    discovered gates — Phase 81 formalizes the `{gates:[...], overall}` rollup; do NOT implement the
    rollup/`overall`/`PASS_WITH_SKIPS` logic here, only leave the per-gate dispatch reusable.

    Gate auto-discovery (D-04): resolve `$gatesDir = Join-Path $PSScriptRoot "gates"`; build the gate
    list with `Get-ChildItem -Path $gatesDir -Filter *.ps1`; gate name =
    `[System.IO.Path]::GetFileNameWithoutExtension($_.Name)`. For a `--gate <name>` run, resolve the
    matching file; if no file matches the requested name, this is a harness-internal error — print a
    diagnostic to stderr and `exit 1` (NOT exit 2, NOT a FAIL verdict — D-05). Dot-source the
    selected gate file (`. $gateFile`) so `Test-Precondition` and `Invoke-Gate` enter scope.

    Dispatch order (D-06): call `Test-Precondition` FIRST. If it returns a non-null string, build a
    verdict object with `verdict = "SKIP_HOST_UNAVAILABLE"` and `reason` = that string, emit it,
    persist it, and `exit 3` — never enter the gate body. If it returns `$null`, call `Invoke-Gate`
    inside a `try/catch`: a thrown exception is a harness-internal error per D-07 (print the message
    to stderr, persist nothing-or-an-error-verdict, `exit 4`) — NEVER catch-and-return PASS, NEVER
    swallow the exception. On normal return, `Invoke-Gate` yields the verdict object.

    Verdict emit (D-01): build with `[ordered]@{}` locking key order `gate, verdict, reason, detail,
    timestamp`; `timestamp` = `Get-Date -Format 'yyyy-MM-ddTHH:mm:ss.fffZ'`; `detail` is a nested
    object so serialize with `ConvertTo-Json -Depth 6 -Compress`; write with
    `[Console]::Out.Write($json + "`n")` to avoid PowerShell CRLF (mirror
    check-upstream-drift.ps1:241-242). Exactly one JSON object is printed per `--gate` run.

    Persistence (D-08): resolve repo root via `Split-Path -Parent $PSScriptRoot`; create
    `.nono-runtime/verdicts/` with `New-Item -ItemType Directory -Force -Path ... | Out-Null`; write
    the same `$json` string to `Join-Path $verdictDir "$Gate.json"` (one file per gate, single source
    of truth). Persist for PASS, FAIL, and SKIP alike.

    Exit mapping (D-02): a single `switch`/if-elseif-else on the verdict string mapping
    `PASS -> exit 0`, `FAIL -> exit 2`, `SKIP_HOST_UNAVAILABLE -> exit 3`; any unexpected verdict
    value is a harness-internal error -> `exit 4`. Reserve 1/4+ so a harness crash or unknown gate
    never reads as a gate FAIL. The runner — not the gate — owns this mapping.

    Do NOT place fenced code blocks in any output; the runner only emits the single compact JSON
    object to stdout (diagnostics go to stderr).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| harness verdict → CI / Phase 81 aggregator | A verdict produced here is trusted by automation to decide milestone completion. A wrong verdict (false PASS) defeats the Dark-Factory mandate. |
| `scripts/gates/*.ps1` glob → dot-source | The runner dot-sources whatever `.ps1` files exist in `scripts/gates/`; those files run with the runner's privileges. |
| `.nono-runtime/verdicts/<gate>.json` → aggregator read | Persisted verdict files are read later (Phase 81) and trusted as the gate outcome. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-76-01 | Tampering/Elevation | runner maps a gate crash or unknown gate to PASS | mitigate | D-07/D-05: `Invoke-Gate` runs in `try/catch`; a throw → `exit 4` harness-internal error, never PASS; unknown `--gate` → `exit 1`, never a FAIL/PASS verdict. Never catch-and-return PASS, never swallow exceptions. |
| T-76-02 | Spoofing/Information | SKIP conflated with PASS at the exit layer | mitigate | D-02: distinct exit codes (PASS=0, FAIL=2, SKIP=3); CI can branch on exit code without parsing JSON, so a host-skip is never read as a pass. |
| T-76-03 | Tampering | `scripts/gates/` glob loads an unexpected/untrusted `*.ps1` | accept | Internal dev tooling (ASVS L1); `scripts/gates/` is repo-tracked source under review. Mitigation is code review + the directory living in the same trust domain as the rest of `scripts/`. No runtime allowlist required at L1. |
| T-76-04 | Tampering (verdict staleness) | aggregator reads a stale `.nono-runtime/verdicts/<gate>.json` from a prior run | accept | Phase 81 (aggregator) re-runs every gate, rewriting each file before reading; Phase 76 only fixes the persistence shape. Each gate writes its own single file (no stale-merge surface). Note for Phase 81 to enforce rewrite-before-read. |
| T-76-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this phase (pure PowerShell tooling). No `[ASSUMED]`/`[SUS]` packages introduced. |
</threat_model>
