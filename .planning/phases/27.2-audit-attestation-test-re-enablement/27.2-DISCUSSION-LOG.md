## Bundle-target architecture (FU-2)

| Option | Description | Selected |
|--------|-------------|----------|
| A — Always mirror to `<audit_root>/<id>/` | Production code change in `rollback_runtime.rs:198-226`: collapse the `if rollback_active` branch and always call `ensure_audit_session_dir`. Single-root verify. Largest behavior change but smallest verify surface. | ✓ |
| B — Dual-location based on `--rollback` (current behavior) | Bundle stays at `<rollback_root>/<id>/...` when `--rollback`, `<audit_root>/<id>/...` otherwise. `cmd_verify` learns dual-root lookup. Most conservative for production. | |
| C — Test rewrite, production unchanged | Smallest production diff. Test 2 renamed to assert `<rollback_root>/<id>` for `--rollback` case. Codifies non-repudiation hole. | |

**User's choice:** A — Mirror to `<audit_root>/<id>/audit-attestation.bundle` regardless of `--rollback`, with backward-compat shim in `cmd_verify` (deletable in v2.5), Plan 22-05a Decision 5 supersession recorded in ADR.

**Notes:** User's framing question — "What one closely mimics the current Linux/macOS functionality?" — surfaced an ambiguity. Investigation of `rollback_runtime.rs:198-226` showed Option B (dual-location) IS the current cross-platform behavior on all platforms (Linux/macOS/Windows). However, the dominant decision driver was security correctness rather than parity preservation: audit attestation must outlive rollback cleanup for non-repudiation, so option A's audit-isolated bundles are the structurally correct target. The current dual-location was characterized in Plan 22-05a Decision 5 as "for backward compatibility" — i.e., a transitional state, not a target state. Option A finishes the audit-isolation migration started by Decision 5.

User added two refinements when locking the decision: (1) ship the v2.5-deletable back-compat shim so legacy sessions remain verifiable, (2) record the Plan 22-05a Decision 5 supersession explicitly in the new ADR so the rationale chain is preserved for future readers.
