<decisions>
## Implementation Decisions

### Bundle-target architecture (FU-2)

- **D-27.2-01 (Locked, Option A):** `nono run --audit-integrity --audit-sign-key <key> [--rollback] -- <cmd>` writes the signed bundle to `<audit_root>/<id>/audit-attestation.bundle` regardless of `--rollback` flag state. Snapshot artifacts (`changes/`, `manifest.json`, etc.) continue to live under `<rollback_root>/<id>/` when `--rollback` is set; only the audit attestation bundle moves.
  - **Why over Option B (current dual-location):** Audit attestation = non-repudiation. Currently `rm -rf ~/.nono/rollbacks/<id>/` deletes the audit bundle along with the snapshot — a non-repudiation hole. Audit bundles must outlive rollback cleanup. Plan 22-05a Decision 5's "for backward compatibility" dual-location rationale is superseded by this phase.
  - **Why over Option C (test rewrite):** Option C codifies the non-repudiation hole and would require renaming `rollback_signed_session_verifies_from_audit_dir_bundle` to assert the wrong intent. The test name is honest under Option A.
  - **Direction-of-travel signal:** Plan 22-05a Decision 5 already split audit-only sessions to `<audit_root>` "for namespace separation"; Option A finishes that migration for rollback-active sessions.

- **D-27.2-02 (Locked, back-compat shim):** `cmd_verify` MUST fall back to `<rollback_root>/<id>/audit-attestation.bundle` when the bundle isn't found at `<audit_root>/<id>/...`, with a one-shot deprecation warning emitted to stderr (`tracing::warn!`) on the fallback hit. Shim is deletable in v2.5; ADR records the v2.5 removal milestone explicitly. This preserves verification of sessions created by older nono versions whose bundles are still at the legacy `<rollback>/<id>` location.

- **D-27.2-03 (Locked, ADR recording):** ADR location: `docs/architecture/audit-bundle-target.md` (mirrors Phase 25-02's `docs/architecture/aipc-unix-futures.md` convention). ADR MUST contain: the option matrix considered (A/B/C with trade-offs), the chosen path with non-repudiation rationale, explicit Plan 22-05a Decision 5 supersession note, the v2.5 shim-removal milestone, and a backward-compat shim contract (when the warning fires, what action it tells the user to take, when the shim deletes).

### Audit-loader swap (FU-1)

- **D-27.2-04 (Locked, cascades from D-27.2-01 + D-27.2-02):** `cmd_verify` in `crates/nono-cli/src/audit_commands.rs` always uses `audit_session::load_session` first. If the audit-side load returns `Session not found`, fall back to `rollback_session::load_session` per the D-27.2-02 shim. No conditional-on-flags loader picking — always-audit-first with fallback chain.
- **D-27.2-05 (Locked):** `#[allow(dead_code)]` removed from `audit_session::load_session` once `cmd_verify` becomes a live consumer. Mirrors Phase 27.1 Plan 02 D-19 pattern of removing the attr when the migration consumer lands.
- **D-27.2-06 (Locked):** Fail-closed surface preserved: when neither loader finds the session, surface the canonical `Session not found` error rather than silently degrading. The shim warning fires only on successful fallback hits, not on full miss.

### Test re-enablement (FU-3) and WR-04/WR-05 cleanups

- **D-27.2-07 (Locked, JSON-shape via Option A cascade):** Test 2's existing assertion `assert!(audit_dir.join("audit-attestation.bundle").exists())` becomes correct by construction once D-27.2-01 lands. No assertion path-rewrite needed for the bundle existence check.
- **D-27.2-08 (Locked, WR-04):** Test 2's JSON-shape assertions (`json["attestation"]["present"]` etc., currently nested) are updated to match `cmd_verify`'s flat shape (`json["attestation_present"]`, `json["attestation_valid"]`, etc.). Production `cmd_verify` JSON output is NOT changed in Phase 27.2. Add a TODO + v2.5 follow-up entry in `deferred-items.md` to nest the JSON when `cmd_verify` ships a v2 schema (Option A's audit-vs-rollback split will likely motivate that schema bump anyway).
- **D-27.2-09 (Locked, WR-05):** Bare `std::env::set_var`/`remove_var` calls in `crates/nono-cli/tests/audit_attestation.rs:347-365` (currently behind localized `#[allow(clippy::disallowed_methods)]`) are replaced with a local RAII guard pattern. The guard restores the prior env-var state on `Drop` so a panic in `run_nono` doesn't leak the test secret env var. Folded into Phase 27.2 since FU-3 is already touching this file.
- **D-27.2-10 (Locked):** Both `#[ignore]` attributes removed from `audit_verify_reports_signed_attestation_with_pinned_public_key` and `rollback_signed_session_verifies_from_audit_dir_bundle`. Final test status target: `cargo test -p nono-cli --test audit_attestation` returns `2 passed; 0 failed; 0 ignored` on Windows host.

### Cross-cutting invariants (carried forward)

- **D-27.2-11 (carried forward from Phase 27.1):** `crates/nono/` stays byte-identical (D-19 invariant). All Phase 27.2 changes live in `crates/nono-cli/src/{audit_commands.rs, rollback_runtime.rs, audit_session.rs}`, `crates/nono-cli/tests/audit_attestation.rs`, and the new ADR file.
- **D-27.2-12 (carried forward from Phase 27.1):** `nono_home_dir()` semantics unchanged. The seam from Phase 27.1 stays as-is.
- **D-27.2-13 (carried forward from Phase 27.1):** `setup_isolated_home` and `run_nono` test helpers stay locked. Phase 27.1 D-27.1-14 Option A pre-creates `<home>/.nono/{audit,rollbacks}` in `setup_isolated_home`; Phase 27.2 keeps that pattern (audit-bundle move means `audit/` will see more traffic, but the dir creation is still needed for `--rollback` snapshot files in `rollbacks/`).

### Claude's Discretion

- **Rollback dir creation when bundle moves out:** Whether `rollback_runtime.rs` still creates `<rollback_root>/<id>/` when `--rollback` is set (for snapshot files) is implementation detail — the planner can keep the existing `create_dir_all` for the rollback path and only redirect `session_dir` for bundle purposes, OR refactor cleanly. Either way as long as snapshot files land under `<rollback_root>/<id>/` and bundle lands under `<audit_root>/<id>/`.
- **Shim warning message wording:** Exact `tracing::warn!` text on shim fallback is the planner's call, but it MUST mention the legacy path, the new canonical path, and the v2.5 deletion milestone.
- **RAII guard implementation:** Whether to write a local `ScopedEnvVar` struct in `audit_attestation.rs` or extract a shared helper in a `tests/common.rs`-style file. Local-only is fine for Phase 27.2 (single test file).

## Existing Code Insights

### Reusable Assets
- **`audit_session::load_session`** (currently `#[allow(dead_code)]`): the audit-aware loader that already understands `<audit_root>/<id>/` layout. FU-1's first-tier loader. Drops `#[allow(dead_code)]` once it has a live consumer (D-27.2-05).
- **`rollback_session::load_session`**: stays in tree for back-compat shim (D-27.2-02). The legacy code path remains correct for sessions created at the old location; it just stops being the primary lookup.
- **`audit_session::ensure_audit_session_dir`**: already used in `rollback_runtime.rs:226` for the audit-only path. FU-2's production change at line 198-226 just collapses the `if rollback_active` branch and always calls `ensure_audit_session_dir`.
- **`Phase 27.1 setup_isolated_home`**: pre-creates `<home>/.nono/{audit,rollbacks}`. Already supports the audit-bundle move; no harness changes needed for FU-2.
- **D-AAH-01 per-invocation env-var pattern** (`{pid}_{nanos}` suffix): used by Test 1 today; D-27.2-09's RAII guard wraps but doesn't replace this pattern.

### Established Patterns
- **`#[allow(dead_code)]` migration pattern (Phase 27.1 D-19):** Add `#[allow(dead_code)]` to a helper when shipping it; remove when the consumer migration lands. Phase 27.2 is the consumer-migration step for `audit_session::load_session`.
- **Back-compat shim with deprecation warning + milestone** (project precedent: Phase 24 upstream-drift, Phase 22 OAuth fallbacks): one-shot `tracing::warn!` on legacy-path hit, ADR documents the removal milestone, code path deletes in the named version.
- **ADR-per-architectural-decision** (Phase 25-02): top-level `docs/architecture/<topic>.md`, formal ADR structure (context / decision / alternatives / consequences). D-27.2-03 follows this convention.

### Integration Points
- **Phase 27.1 NONO_TEST_HOME seam:** Phase 27.2 does NOT modify `crates/nono-cli/src/config/mod.rs`. The seam reaches the supervisor unchanged; only the supervisor's bundle-target routing changes.
- **`audit verify --json` consumers:** Phase 27.2's D-27.2-08 keeps the existing flat JSON shape. Any external tooling reading `attestation_present`/`attestation_valid` keeps working. v2.5 v2-schema follow-up will deprecate flat keys with notice.
- **Phase 22-05a/b cryptographic primitives:** `sign_session_attestation` / `verify_audit_attestation` signatures unchanged. Bundle filename constant (`ATTESTATION_BUNDLE_FILENAME = "audit-attestation.bundle"`) unchanged. Only the `<session_dir>` derivation upstream changes; the bundle write/read code paths are agnostic.
