<objective>
Translate the Unix supervisor IPC hardening intent onto the fork's Windows Named-Pipe AIPC path (Phase 18) — SC1 + SC4. Replace the unbounded `read_exact` in `socket_windows.rs::read_frame` with a `PeekNamedPipe` poll-until-data-or-deadline bounded read (D-03), and convert the capability-pipe server loop in `exec_strategy_windows/supervisor.rs` from hard-break-on-close to a re-accept loop (`DisconnectNamedPipe` + `ConnectNamedPipe`) bounded by `terminate_requested`. Add Windows-gated integration tests (driving the private `read_frame`/`read_exact_bounded` via the reachable pub `recv_message()` path) and a documented live-repro UAT (operator-run on a real Win11 console).

Purpose: Windows AIPC pipes are `PIPE_WAIT` (blocking) and do not honor socket-style read timeouts, and a single transient capability-pipe close currently disables capability expansion for the whole session. This is a genuinely divergent transport (named pipe, not AF_UNIX) — D-03 mandates the conservative `PeekNamedPipe` poll over an overlapped-I/O rewrite (explicitly rejected as too large/risky), and SC4 REQUIRES the SUMMARY document this as a translate-not-cherry-pick with the rejected-alternative rationale.

TEST REACHABILITY (Blocker fix): `nono-cli` is BIN-ONLY (no `[lib]`). The bounded `read_frame`/`read_exact_bounded` are PRIVATE to the `nono` LIBRARY. Integration tests in `crates/nono-cli/tests/` exercise them via the LIBRARY pub surface — `nono::supervisor::socket::SupervisorSocket::recv_message()` drives the private `read_frame` — never `nono_cli::` and never a direct `read_frame`/`read_exact_bounded`/`run_supervisor_loop` reference.

Output: Modified `socket_windows.rs` (bounded `read_frame`), `exec_strategy_windows/supervisor.rs` (re-accept loop), Windows-gated tests in `supervisor_ipc_robustness_windows.rs`, and a Windows live-repro UAT documented in the SUMMARY.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| sandboxed child → supervisor AIPC named pipe | Untrusted child controls pipe frame timing/content + connect/disconnect cadence; the read + accept paths are the attack surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-59-03a | Denial of Service | `socket_windows.rs::read_frame` partial-frame hang under PIPE_WAIT | mitigate | `PeekNamedPipe` deadline poll bounds the read; a slow/silent child cannot block indefinitely (Task 1; verified by the Windows bounded-read test via `recv_message()` + live-repro SC2). |
| T-59-03b | Denial of Service | PeekNamedPipe busy-spin pegging a CPU core | mitigate | `std::thread::sleep(~10ms)` between peeks (Pitfall 2); bounded by the deadline. |
| T-59-03c | Denial of Service | reconnect thrash on the capability pipe | mitigate | Re-accept loop bounded by `terminate_requested` + child liveness — reconnect cannot outlive the child; no infinite tight loop, no separate retry cap needed (fail-secure). |
| T-59-03d | Spoofing | reconnect impersonation (different principal connects after re-accept) | mitigate | Re-accept re-verifies the per-session SID/token on every connect; trust is NOT cached across reconnect (V3); the restricting-SID DACL on the pipe is unchanged. |
| T-59-03e | Elevation / Tampering | timeout/disconnect induces fail-open (granting a capability on a partial request) | mitigate | A timed-out/partial/malformed frame is ABANDONED; `seen_request_ids` replay protection preserved across re-accept; no capability granted on a partial/timed-out request (fail-closed Security Domain invariant). |
| T-59-03f | Tampering | re-accepting wrong instance count on a 1-instance control pipe | mitigate | Re-arm the EXISTING handle (`DisconnectNamedPipe`+`ConnectNamedPipe`) for control pipes rather than creating a fresh instance (Pitfall 3); avoids ERROR_PIPE_BUSY/handle leak. |
| T-59-SC | Tampering | npm/pip/cargo installs | accept | No package installs — RESEARCH.md §"Package Legitimacy Audit": zero new deps (vendored `windows-sys 0.59`, `std` only). No `[ASSUMED]`/`[SUS]` packages → no legitimacy checkpoint. |
</threat_model>
