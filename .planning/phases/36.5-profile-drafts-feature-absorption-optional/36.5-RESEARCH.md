## Summary

Phase 36.5 absorbs upstream commit `829c341a` (Luke Hinds / v0.47.1, +796 / -21 across 10 files). The CONTEXT.md decisions D-36.5-A1..D4 are locked, but research surfaced **eight material deltas between upstream's actual shape and the decisions**, all of which need to be accounted for by the planner. Most importantly:

1. **Upstream `ActionRequired` is a single-field tuple variant `ActionRequired(String)`, NOT the `{ expected, actual, resolve_via }` struct CONTEXT.md prescribes in D-36.5-B2.** D-36.5-B2 is a deliberate fork-divergence — the planner needs to accept this as a knowing departure from upstream (which it already is via the "typed-struct" wording) and write tasks that build the struct shape from scratch, not via cherry-pick. This pushes Commit 2 to D-20 manual-replay, not D-19.
2. **Upstream has NO `nono profile init --draft` subcommand.** Drafts are simply files dropped into `~/.config/nono/profile-drafts/` (likely by external tooling or hand). D-36.5-A1's `init --draft` shape is a fork invention designed to give the user a first-party create path. The `--refresh` mode mentioned in D-36.5-B2's `resolve_via` string is *also* fork-invented; upstream has no analog. Research recommendation: include `--refresh` in Commit 1 as CONTEXT.md suggests.
3. **Upstream stores the base hash in a sibling `<name>.base` file**, NOT inside the draft JSON. Two new helpers (`get_user_profile_draft_path`, `get_user_profile_draft_base_path`) live in `profile/mod.rs`, not in `package.rs`. CONTEXT.md D-36.5-B1 says "Add `profile_drafts_dir()` in `package.rs` mirroring `profiles_dir()`" — but upstream actually puts the helpers in `profile/mod.rs` alongside `get_user_profile_path`. The planner should decide whether to mirror upstream's location or follow D-36.5-B1's `package.rs` placement; either works but they're inconsistent.
4. **`is_package_managed_profile` does NOT exist in fork.** CONTEXT.md references `profile/mod.rs:1932`, but actual line 1936 has `get_package_for_profile(name) -> Option<PathBuf>` (returns `Some(package_dir)` if package-managed, `None` otherwise). Upstream's `reserved_profile_source` uses `find_pack_store_profile(name)` which **also does not exist in fork** (explicit fork-divergence at `profile/mod.rs:2358`). The planner must use `get_package_for_profile` (or its underlying `is_profile_symlink_into_package_store`) for the package-managed shadow check.
5. **`would_shadow_builtin` already exists at `profile_save_runtime.rs:227`.** It checks both `is_user_override` AND `policy.profiles.contains_key`. The planner can reuse this directly for the built-in shadow check in D-36.5-D1 — no new helper needed for built-in detection.
6. **The cherry-pick probe surfaced 533 LOC across 10 hunks in 6 files — well below the 500 LOC threshold for the bulk of the conflict surface, but the cumulative total is 533.** Per-file breakdown shows all conflicts are *mechanical* (additive blocks at shifted anchors, plus one structurally-divergent hunk in `profile_runtime.rs` where fork has Linux-cfg-gated `pre_create_landlock_profiles_dir()` helper vs upstream's inline `create_dir_all` block). Recommendation: D-20 manual-replay for the whole absorption, NOT D-19 cherry-pick. Rationale below.
7. **Upstream uses the `similar` crate for promote-diff rendering** — already at `crates/nono-cli/Cargo.toml:48` as `similar = "2"`. Zero workspace dep churn.
8. **Upstream uses `nix::libc::O_NOFOLLOW` for symlink-safe reads in `read_regular_file()`** — fork has `nix` at `crates/nono-cli/Cargo.toml:86` but ONLY behind `#[cfg(unix)]`. The fork's port needs cross-platform shape; on Windows there's no `O_NOFOLLOW` analog, so the planner must either gate the noflw-flag to `#[cfg(unix)]` or use a different symlink-safe pattern (e.g., `symlink_metadata` precheck, which upstream's `regular_file_exists` already does).

**Primary recommendation:** D-20 manual-replay path for the whole absorption (all 3 commits). The 533 LOC conflict surface is below the 500 LOC threshold for "mechanical," but the eight fork-divergence deltas mean even a successful D-19 cherry-pick would require immediate follow-up commits to convert the struct shape, the sidecar `.base` file, the `--draft` subcommand, and the cross-platform `O_NOFOLLOW` gate. D-20 commits cite `829c341a` as design source without the trailer.

## Atomic-Write Semantics (Windows)

**Pattern in fork at `package.rs:309-320` (verified):**
```rust
let tmp_path = path.with_extension("json.tmp");
fs::write(&tmp_path, format!("{json}\n")).map_err(NonoError::Io)?;
fs::rename(&tmp_path, &path).map_err(NonoError::Io)?;
```

**Upstream's pattern in `profile_cmd.rs::atomic_write_file` (verified):**
- Writes to `parent.join(format!(".{filename}.tmp.{pid}", pid = std::process::id()))` (dotfile prefix + pid suffix for concurrency).
- Calls `file.sync_all()` before drop (forces fsync; durability guarantee).
- `fs::rename(&tmp_path, &path)` after close.
- Cleanup via `fs::remove_file(&tmp_path)` on any error path.

**Windows-specific semantics (sourced from std::fs::rename docs):**
- `fs::rename` on Windows is implemented via `MoveFileExW` with `MOVEFILE_REPLACE_EXISTING`. It atomically replaces the destination if it exists, **provided both source and target are on the same volume**. Cross-volume rename is NOT atomic (becomes copy+delete with a window of inconsistency).
- **Same-volume cross-directory rename** (e.g., `~/.config/nono/profile-drafts/myagent.json` → `~/.config/nono/profiles/myagent.json`) is atomic. Both are under `%APPDATA%\nono\` which lives on the same volume by definition.
- **Within same directory** (e.g., `<draft>.json.tmp` → `<draft>.json` for the init-draft create flow) is always atomic.
- **Antivirus / Windows Defender** can hold open handles on freshly-written files briefly, causing intermittent `ERROR_SHARING_VIOLATION` (`os error 32`) on `fs::rename`. This is a known issue but rare for sub-MB profile JSON. The fork's existing `write_lockfile` uses the same idiom without reported issues across Phases 22-04.
- **Filesystem case sensitivity:** Windows is case-insensitive; the upstream test `yanked_message_pins_latest_when_available` uses `eq_ignore_ascii_case` for hash comparison — this is `read_regular_file` content, not path, so case insensitivity doesn't affect it. Path comparisons in shadow detection must NOT use lowercased paths (per CLAUDE.md path-security note).

**Recommendation for the planner:**
- **Commit 1 (init --draft):** Reuse the fork's `package.rs:309-320` pattern verbatim — `path.with_extension("json.tmp")` + `fs::write` + `fs::rename`. Same idiom.
- **Commit 2 (promote):** Port upstream's `atomic_write_file` helper directly. The `.<name>.tmp.<pid>` pattern is more defensive (handles concurrent promotes of different profiles, distinct PIDs) and includes `sync_all()` which the fork's lockfile path does NOT. Worth adopting for the cross-directory rename `profile-drafts/myagent.json` → `profiles/myagent.json` since that's a more critical operation than lockfile.
- **Cross-directory rename within `%APPDATA%\nono\`:** No special handling needed; same volume. Document in commit body.
- **No need to gate antivirus retry logic** in scope — no operational evidence of issues, and the planner already locked SHA-256 base-hash verification which would catch any post-rename corruption.

## Shadow-Detection Surface

### Built-in shadow check

**Existing helper:** `profile_save_runtime.rs:227 fn would_shadow_builtin(profile_name: &str) -> bool`. Body:

```rust
fn would_shadow_builtin(profile_name: &str) -> bool {
    if profile::is_user_override(profile_name) {
        return false;
    }
    match crate::policy::load_embedded_policy() {
        Ok(policy) => policy.profiles.contains_key(profile_name),
        Err(_) => true,  // fail-closed: refuse to save if policy load fails
    }
}
```

**Promote semantics differ from save semantics:**
- `would_shadow_builtin` returns `false` if user already overrode (because user already chose to shadow, so a subsequent save is just an update).
- For promote, the same semantic holds — promoting over an existing user profile (even one that shadows a built-in) is fine; the user has already taken the shadow action. Promoting OVER a built-in with NO existing user profile is the case to refuse.
- Conclusion: `would_shadow_builtin` is **directly reusable** in `cmd_promote` for the built-in shadow check. Visibility is `fn` (module-private); planner needs to either (a) make it `pub(crate)` or (b) write an inline equivalent in `profile_cmd.rs`. Recommendation: **promote to `pub(crate)`** since profile_save and promote share the same shadow semantic. One source of truth.

### Package-managed shadow check

**No `is_package_managed_profile` exists.** Use one of:

- **`profile::get_package_for_profile(name) -> Option<PathBuf>`** (`profile/mod.rs:1936`). Returns `Some(package_dir)` if name resolves to a symlink into the package store. Treat `Some(_)` as "package-managed." This is the highest-level API — recommended.
- **`package::is_profile_symlink_into_package_store(profile_name) -> Option<PathBuf>`** (`package.rs:339`). The underlying primitive. Same return shape. `get_package_for_profile` is just a thin wrapper that adds a profile-name validity check first.

### Embedded built-in profile table — clarification

CONTEXT.md says "embedded built-in profile data + name table for shadow-detection" referencing `config/embedded.rs`. Research finding: there is no separate "name table." The embedded built-in profile data is the entire `policy.json` blob (loaded via `embedded::embedded_policy_json()` → parsed via `policy::load_embedded_policy()` → produces `Policy { profiles: HashMap<String, ProfileDef>, ... }`). Built-in name membership = `policy.profiles.contains_key(name)`. The four named profiles in CONTEXT.md (`claude-code`, `codex`, `opencode`, `claude-no-keychain`) are entries in this map, NOT a hardcoded Rust array.

`would_shadow_builtin` already does this check correctly. No new helper needed.

### Helper-shape recommendation (D-36.5-D1 Claude's Discretion)

CONTEXT.md gives the planner discretion to choose:
- (a) inline the two checks directly in `cmd_promote`, or
- (b) write a single `is_shadowed_profile(name) -> Result<Option<&'static str>>` wrapper in `profile_cmd.rs`.

Research **recommendation: (b) — single wrapper**, modeled after upstream's `reserved_profile_source` (which returns `Option<&'static str>` for "built-in" vs "installed pack" labels). Reasoning:
1. The wrapper return shape (`Option<&'static str>`) lets `cmd_promote` build a context-aware error message ("built-in" vs "package-managed" → different `resolve_via` strings per D-36.5-D3).
2. Future `validate --draft` extensions might want the same check.
3. Mirrors upstream's structure, which makes the D-20 manual-replay map cleaner.

**Recommended signature:**
```rust
fn reserved_profile_source(name: &str) -> Result<Option<&'static str>> {
    let pol = policy::load_embedded_policy()?;
    if pol.profiles.contains_key(name) && !profile::is_user_override(name) {
        return Ok(Some("built-in"));
    }
    if profile::get_package_for_profile(name).is_some() {
        return Ok(Some("package-managed"));
    }
    Ok(None)
}
```
This is upstream's helper minus the `find_pack_store_profile` call (which doesn't exist in fork), substituting `get_package_for_profile`.

## `--refresh` Mode Investigation

**Upstream behavior:** Upstream commit `829c341a` does **NOT** include a `--refresh` flag on any subcommand. There is no `init --draft` subcommand in upstream at all. The draft create flow in upstream is implicit — users drop files into `~/.config/nono/profile-drafts/` by whatever means (text editor, scripts, external tools). The `.base` sidecar file is also created by whatever upstream tool generates the draft.

D-36.5-B2's `resolve_via: nono profile init --draft --refresh <name>` string is **entirely fork-invented**, designed to give operators a first-party way to reconcile a stale base-hash. Without `--refresh`, an operator hitting `ActionRequired` would have to manually delete the draft and re-create.

**Recommended semantic for `--refresh`:**

`nono profile init --draft --refresh <name>` should:
1. Verify `<name>` already exists in `profile-drafts/` (otherwise error "no draft to refresh; use `init --draft` without `--refresh`").
2. Read the current canonical profile at `profiles/<name>.json` (error if missing → "nothing to refresh from").
3. Recompute the base-hash from the current canonical bytes.
4. Re-write the draft on disk:
   - If the canonical profile differs from the draft → preserve the draft's content (don't clobber user edits), update only the recorded base-hash.
   - If the canonical and draft are identical → fast path, just update base-hash.
5. Whether the base-hash lives in a sidecar `.base` file (upstream shape) or as a JSON field inside the draft (a fork-divergent shape) is a planner decision. Recommendation: **adopt upstream's sidecar `.base` file** so promote-time hash extraction is a separate file read; this lets the draft JSON remain a valid Profile schema without a synthetic top-level `_base_hash` field that would fail strict JSON Schema validation.

**Recommendation for planner:** **Include `--refresh` in Commit 1** as the Claude's-Discretion guidance suggests. Reasoning:
1. The `ActionRequired` Display message in D-36.5-B2 hardcodes a reference to `nono profile init --draft --refresh <name>` — leaving the flag out means the error message points at a non-existent command.
2. `--refresh` is mechanically small (one new bool field on `ProfileInitArgs`, one branch in `cmd_init`, ~40 LOC of logic).
3. Defers create complexity to its own dedicated test surface.

**Alternative (defer to v2.5-FU):** Change the `resolve_via` message to "regenerate the draft" without specifying a command. This is operationally weaker but keeps Phase 36.5 narrower.

### Pattern 1: Atomic Write
**What:** Write to sibling `.tmp` file, fsync, rename to final path.
**When to use:** Any create/update of an on-disk profile artifact where partial writes would corrupt downstream consumers.
**Example (verbatim from `package.rs:309-320`):**
```rust
// Source: crates/nono-cli/src/package.rs:309-320
pub fn write_lockfile(lockfile: &Lockfile) -> Result<()> {
    let path = lockfile_path()?;
    if let Some(parent) = path.parent() {
        fs::create_dir_all(parent).map_err(NonoError::Io)?;
    }
    let tmp_path = path.with_extension("json.tmp");
    let json = serde_json::to_string_pretty(lockfile)
        .map_err(|e| NonoError::ConfigParse(format!("failed to serialize lockfile: {e}")))?;
    fs::write(&tmp_path, format!("{json}\n")).map_err(NonoError::Io)?;
    fs::rename(&tmp_path, &path).map_err(NonoError::Io)?;
    Ok(())
}
```

### Pattern 2: Atomic Write with PID-suffixed Tempfile + fsync
**What:** Defensive variant — handles concurrent operations + durability.
**When to use:** Promote operation (cross-directory atomic rename across user-visible artifacts).
**Example (from upstream 829c341a `profile_cmd.rs::atomic_write_file`, adapted for fork cross-platform shape):**
```rust
// Source: upstream 829c341a profile_cmd.rs::atomic_write_file (D-20 manual-replay)
fn atomic_write_file(path: &Path, contents: &[u8]) -> Result<()> {
    let parent = path.parent().ok_or_else(|| /* error */)?;
    fs::create_dir_all(parent)?;
    let file_name = path.file_name().ok_or_else(|| /* error */)?;
    let mut tmp_name = std::ffi::OsString::from(".");
    tmp_name.push(file_name);
    tmp_name.push(format!(".tmp.{}", std::process::id()));
    let tmp_path = parent.join(tmp_name);
    let mut file = fs::OpenOptions::new()
        .write(true).create(true).truncate(true)
        .open(&tmp_path)?;
    file.write_all(contents)?;
    file.sync_all()?;  // fsync before rename
    drop(file);
    fs::rename(&tmp_path, path)?;
    Ok(())
}
```

### Pattern 3: Symlink-Safe File Read
**What:** Open with `O_NOFOLLOW` on Unix; symlink_metadata precheck on all platforms.
**When to use:** Reading attacker-controllable file paths (e.g., a draft path that an attacker may have planted a symlink at).
**Example (cross-platform shape, adapted for fork):**
```rust
fn read_regular_file(path: &Path, label: &str) -> Result<Vec<u8>> {
    // Cross-platform precheck via symlink_metadata
    let meta = fs::symlink_metadata(path)?;
    if meta.file_type().is_symlink() {
        return Err(NonoError::ProfileParse(format!("{label} must not be a symlink: {}", path.display())));
    }
    if !meta.file_type().is_file() {
        return Err(NonoError::ProfileParse(format!("{label} is not a regular file: {}", path.display())));
    }
    // Unix: also defend against TOCTOU via O_NOFOLLOW (cross-platform shape; Windows skips)
    let mut options = fs::OpenOptions::new();
    options.read(true);
    #[cfg(unix)]
    {
        use std::os::unix::fs::OpenOptionsExt;
        options.custom_flags(nix::libc::O_NOFOLLOW);
    }
    let mut file = options.open(path)?;
    let mut bytes = Vec::new();
    file.read_to_end(&mut bytes)?;
    Ok(bytes)
}
```

### Anti-Patterns to Avoid
- **In-place mutation of `profiles/<n>.json`:** Always write to `.tmp` + rename. Mid-write crash leaves a half-written canonical profile.
- **`path.to_string_lossy().starts_with(...)`** for shadow-detection: use `Path::starts_with()` per CLAUDE.md path-security note.
- **`unwrap()` on `policy::load_embedded_policy()`:** fail-closed (refuse promote) per CLAUDE.md.
- **Trusting `args.name` without `is_valid_profile_name`:** existing check rejects `../`, `/`, `.json`, etc. Call it first.
- **Skipping base-hash check on `--yes`:** `--yes` only bypasses the interactive prompt, NOT the base-hash verification. Both checks fail-closed.
- **Reusing `NonoError::ProfileParse` for shadow-refusal:** must be `NonoError::ActionRequired` (D-36.5-D3) so the C FFI consumer can pattern-match on the typed variant.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SHA-256 hashing | Manual block-process loop | `sha2::Sha256::digest(bytes)` | Side-channel-resistant impl; FIPS-ready. Already in deps. |
| Hex encoding | Manual nibble loop | `format!("{byte:02x}")` per byte (upstream pattern) OR `hex::encode` | Tiny perf delta; format-string approach matches upstream verbatim. |
| Unified diff rendering | Manual line-by-line scan | `similar::TextDiff::from_lines` + `ChangeTag` matching | Handles context, hunks, and ANSI color. Already in deps. |

## Common Pitfalls

### Pitfall 1: `ActionRequired` shape divergence between fork and upstream
**What goes wrong:** D-36.5-B2 prescribes `ActionRequired { expected, actual, resolve_via }` struct shape. Upstream `829c341a` has `ActionRequired(String)` single-tuple. If the planner naively cherry-picks upstream's call sites (`Err(NonoError::ActionRequired(yanked_message(...)))`), they'd fail to compile against the fork's struct variant.
**Why it happens:** D-36.5-B2 is a deliberate fork-divergence (typed variant is more pattern-match-friendly), but the upstream code uses single-tuple everywhere.
**How to avoid:** D-20 manual-replay (recommended). For each upstream callsite, convert from `ActionRequired(String)` to `ActionRequired { expected: ..., actual: ..., resolve_via: ... }`. The `expected`/`actual` fields are most natural for base-hash mismatch; for shadow-refusal and package-status advisory, embed the multi-line message in `resolve_via` and use empty strings or context-specific markers in `expected`/`actual`. Document the convention in `error.rs`'s docstring on the variant.
**Warning signs:** Compile error `expected struct fields, found String` on `ActionRequired(...)` calls. Quick fix: never tuple-construct.

### Pitfall 2: `find_pack_store_profile` doesn't exist in fork
**What goes wrong:** Upstream's `reserved_profile_source` helper calls `profile::find_pack_store_profile(name)`. Direct port produces a compile error.
**Why it happens:** Fork explicitly omitted this function during Plan 34-04 (documented at `profile/mod.rs:2358`).
**How to avoid:** Use `profile::get_package_for_profile(name).is_some()` instead. Same semantic (returns `Option<PathBuf>` to package dir).
**Warning signs:** `cannot find function 'find_pack_store_profile' in module 'profile'`.

### Pitfall 3: `nono::libc::O_NOFOLLOW` is Unix-only
**What goes wrong:** Upstream's `read_regular_file` calls `options.custom_flags(nix::libc::O_NOFOLLOW)`. `nix::libc` is Unix-only; on Windows this crate import fails.
**Why it happens:** Upstream commit assumes Unix-only build; fork must compile on all three platforms.
**How to avoid:** Gate `#[cfg(unix)]` around the `custom_flags` call. Rely on cross-platform `symlink_metadata` precheck in `regular_file_exists` for Windows.
**Warning signs:** Windows clippy fails with `unresolved import nix::libc::O_NOFOLLOW`. The fork's `crates/nono-cli/Cargo.toml:86` already gates `nix` to Unix-only — confirms the planner needs the `#[cfg(unix)]` guard.

### Pitfall 4: `profile-drafts/` dir auto-creation in `profile_runtime.rs` clashes with fork's `pre_create_landlock_profiles_dir`
**What goes wrong:** Upstream adds an inline cross-platform `create_dir_all` block for both `profiles/` and `profile-drafts/`. Fork has a Linux-cfg-gated `pre_create_landlock_profiles_dir()` helper from Plan 36 (REQ-PORT-CLOSURE-06) that does only `profiles/`. Naive merge either double-creates `profiles/` on Linux or skips drafts on non-Linux.
**Why it happens:** Fork's Landlock helper is Linux-only by design (Landlock is Linux-only); drafts dir creation needs cross-platform.
**How to avoid:** Two-layer fix:
1. Keep `pre_create_landlock_profiles_dir()` for Linux profiles-dir.
2. Add a separate `pre_create_drafts_dir()` helper (cross-platform; best-effort `create_dir_all`) called unconditionally in `prepare_profile_with_options`.
**Warning signs:** Linux clippy compiles but Windows produces missing-drafts-dir at runtime; OR Linux double-mkdir warning under `RUST_LOG=trace`.

### Pitfall 5: Base-hash sidecar `.base` file vs JSON field
**What goes wrong:** D-36.5-B4 says "SHA-256 base-hash" without specifying storage location. Two patterns:
  - **(a) Sidecar:** `profile-drafts/<name>.base` containing 64-char hex string (upstream pattern).
  - **(b) Inline:** A `_base_hash` field inside the draft JSON.
Inline (b) makes the draft fail JSON Schema validation (extra field).
**Why it happens:** CONTEXT.md doesn't disambiguate.
**How to avoid:** Adopt upstream's sidecar pattern. Two helpers: `get_user_profile_draft_path` returns `profile-drafts/<n>.json`; `get_user_profile_draft_base_path` returns `profile-drafts/<n>.base`. Atomic-write both files at init/refresh time; read both at promote time.
**Warning signs:** JSON Schema validation fails on drafts when `--strict` is set; OR the planner ends up adding `_base_hash` to the canonical `Profile` struct (cross-contaminating canonical schema).

### Pitfall 6: Windows-host clippy passes; Linux/macOS cross-target clippy fails on unused imports
**What goes wrong:** New code in `profile_cmd.rs` (`read_regular_file`, `atomic_write_file`) imports `std::os::unix::fs::OpenOptionsExt` for `custom_flags`. Unix-cfg-gating the use site without cfg-gating the import produces `unused_imports` warning on Windows-host clippy.
**Why it happens:** `cargo clippy` on Windows host doesn't compile Unix-cfg blocks, so the import looks unused; on Linux/macOS host, the import is used and OK.
**How to avoid:** Cfg-gate BOTH the import and the use site: `#[cfg(unix)] use std::os::unix::fs::OpenOptionsExt;` at the top of the function block, or scope the import inside the `#[cfg(unix)] { ... }` block.
**Warning signs:** Phase 25 CR-A regression lesson (per `memory/feedback_clippy_cross_target.md`). Both `--target x86_64-unknown-linux-gnu` and `--target x86_64-apple-darwin` clippy gates (D-36.5-C4 gates 3 + 4) catch this.

### Pitfall 7: Test concurrency on `HOME` / `XDG_CONFIG_HOME` / `APPDATA`
**What goes wrong:** Multiple tests setting env vars in parallel produces flaky test failures (CLAUDE.md § Environment variables in tests).
**Why it happens:** Rust unit tests run in parallel within the same process by default.
**How to avoid:** Use `crate::test_env::ENV_LOCK` global mutex (already established pattern; see `profile_save_runtime.rs` tests). Use `EnvVarGuard::set_all(&[("XDG_CONFIG_HOME", ...)])` to save/restore. Keep modified window as short as possible.
**Warning signs:** Test passes alone, fails in `cargo test --workspace`. Look for env-var manipulation without `ENV_LOCK`.

### Pitfall 8: `--yes` doesn't skip the diff render
**What goes wrong:** Operator running `nono profile promote myagent --yes` in CI sees a 200-line diff dumped to stdout, which clutters logs.
**Why it happens:** Upstream renders diff before the prompt unconditionally.
**How to avoid:** Either (a) skip diff render when `args.yes == true`, OR (b) emit a one-line summary instead ("Promoting draft: 3 added, 1 removed, 5 changed"). Recommendation: (b) — operators still want some signal of what changed.
**Warning signs:** CI logs explode after first promote landing. Add a test asserting `--yes` output is < N lines.

fn read_regular_file(path: &Path, label: &str) -> Result<Vec<u8>> {

### Verifying base-hash (sidecar pattern)
```rust
// Source: adapted from upstream 829c341a profile_cmd.rs::verify_base_hash
// D-20 manual-replay; mapped to fork's ActionRequired struct shape.
fn verify_base_hash(base_path: &Path, current_bytes: &[u8]) -> Result<()> {
    let base_bytes = read_regular_file(base_path, "profile draft base hash")?;
    let provided = std::str::from_utf8(&base_bytes)
        .map_err(|e| NonoError::ProfileParse(format!("base hash is not UTF-8: {e}")))?
        .trim();
    if provided.len() != 64 || !provided.chars().all(|c| c.is_ascii_hexdigit()) {
        return Err(NonoError::ProfileParse(format!(
            "invalid base hash in {}",
            base_path.display()
        )));
    }
    let current = sha256_hex(current_bytes);
    if !provided.eq_ignore_ascii_case(&current) {
        return Err(NonoError::ActionRequired {
            expected: provided.to_string(),
            actual: current,
            resolve_via: format!("nono profile init --draft --refresh <name>"),  // placeholder; cmd_promote fills with args.name
        });
    }
    Ok(())
}

fn sha256_hex(bytes: &[u8]) -> String {
    use sha2::{Digest, Sha256};
    let digest = Sha256::digest(bytes);
    digest.iter().map(|byte| format!("{byte:02x}")).collect()
}
```

### Shadow-detection wrapper

fn reserved_profile_source(name: &str) -> Result<Option<&'static str>> {
    let pol = crate::policy::load_embedded_policy()?;

### Atomic cross-directory rename (promote)
```rust
// Source: upstream 829c341a profile_cmd.rs::atomic_write_file pattern.
// Verbatim port — fork's Windows fs::rename is atomic on same-volume.
fn atomic_write_file(path: &Path, contents: &[u8]) -> Result<()> {
    let parent = path.parent().ok_or_else(|| {
        NonoError::ProfileParse(format!("invalid profile path {}", path.display()))
    })?;
    fs::create_dir_all(parent).map_err(|e| NonoError::ProfileRead {
        path: parent.to_path_buf(),
        source: e,
    })?;
    let file_name = path.file_name().ok_or_else(|| {
        NonoError::ProfileParse(format!("invalid profile path {}", path.display()))
    })?;
    let mut tmp_name = std::ffi::OsString::from(".");
    tmp_name.push(file_name);
    tmp_name.push(format!(".tmp.{}", std::process::id()));
    let tmp_path = parent.join(tmp_name);
    let mut file = fs::OpenOptions::new()
        .write(true).create(true).truncate(true)
        .open(&tmp_path)
        .map_err(|e| NonoError::ProfileRead { path: tmp_path.clone(), source: e })?;
    if let Err(error) = file.write_all(contents) {
        let _ = fs::remove_file(&tmp_path);
        return Err(NonoError::ProfileRead { path: tmp_path, source: error });
    }
    if let Err(error) = file.sync_all() {
        let _ = fs::remove_file(&tmp_path);
        return Err(NonoError::ProfileRead { path: tmp_path, source: error });
    }
    drop(file);
    if let Err(error) = fs::rename(&tmp_path, path) {
        let _ = fs::remove_file(&tmp_path);
        return Err(NonoError::ProfileRead { path: path.to_path_buf(), source: error });
    }
    Ok(())
}
```

## Open Questions (RESOLVED)

All five questions resolved during planning. Resolutions adopted in `36.5-01-DRAFTS-ABSORPTION-PLAN.md`:

1. **`NonoError::ActionRequired` shape: how to map upstream's single-String message to the typed struct?**
   - What we know: D-36.5-B2 prescribes `{ expected, actual, resolve_via }` for base-hash mismatch. Upstream uses `ActionRequired(String)` everywhere.
   - What's unclear: For shadow-refusal (D-36.5-D3) and package-status-yanked (upstream's `enforce_official_pack_status`), there's no natural "expected/actual" pair. Should those callsites use `expected: "".to_string(), actual: "".to_string(), resolve_via: full_message` (empty markers) or should the variant grow a variant-of-variants shape?
   - RESOLVED: Document an explicit convention in the variant's docstring: for base-hash, fields are literal hex; for shadow / advisory, `expected`/`actual` are empty strings (or marker constants like `"<not applicable>"`) and `resolve_via` carries the multi-line operator message. Reconsider only if a third callsite shape emerges. **Adopted in plan must_haves.truths (3-row convention table).**

2. **`--refresh` flag scope: should it also work without an existing canonical?**
   - What we know: D-36.5-B2 mentions `--refresh` for stale-base-hash recovery.
   - What's unclear: If `profiles/<name>.json` doesn't exist (first-time draft), should `--refresh` no-op (since there's no canonical to refresh against), error, or wipe the recorded base-hash?
   - RESOLVED: error with a clear message ("no canonical profile to refresh against; the draft is already a first-time promote candidate"). Avoids surprising state mutations. **Adopted in plan Task C1-02 Test 5.**

3. **`package_status.rs` enforcement-timing: at every `prepare_profile_with_options` call?**
   - What we know: Upstream wires `enforce_for_active_profile` unconditionally after `load_profile`.
   - What's unclear: Does the network round-trip add unacceptable latency on every `nono run`? Should there be a cache (e.g., once per process, or TTL-cached) or an opt-out env var?
   - RESOLVED: Match upstream behavior (every call) for Phase 36.5; defer cache + opt-out to v2.5-FU if user reports surface. Document the network round-trip in `docs/cli/features/profiles-groups.mdx`. **Adopted in plan Task C3-03.**

4. **C FFI Display string format vs typed-struct shape consistency.**
   - What we know: D-36.5-B3 says map to existing `ErrConfigParse`. The C consumer pattern-matches on `nono_last_error()` Display string if it needs to distinguish base-hash from shadow.
   - What's unclear: How stable is the Display string format across versions? If a future plan changes the Display wording, C consumers break silently.
   - RESOLVED: Document in `bindings/c/src/types.rs` that `ErrConfigParse` Display strings are operator-facing, NOT a stable API for C consumers. C consumers wanting structured discrimination must use a future `nono_last_error_kind()` or similar. Mark `<future-FU>`. **Adopted in plan Task C2-01 (binding docs note).**

5. **Plan 22-03 PKG-02 `%LOCALAPPDATA%` doc cleanup placement.**
   - What we know: Deferred to v2.5-FU-N per CONTEXT.md Deferred section.
   - What's unclear: Whether to opportunistically fix it in Phase 36.5 Commit 3 docs touch (CONTEXT.md says "may include opportunistically").
   - RESOLVED: Don't opportunistically fix in Commit 3 — keeps PR boundary clean. Register as a v2.5-FU-N if no docs cleanup phase exists. **Adopted in plan scope (Commit 3 docs surface explicitly excludes 22-03 cleanup).**
