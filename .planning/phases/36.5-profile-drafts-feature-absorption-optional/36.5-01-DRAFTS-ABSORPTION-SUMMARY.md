# Phase 36.5 Plan 01: Drafts Absorption Summary

**Profile-drafts review workflow absorbed from upstream 829c341a as 3-commit D-20 manual-replay: `init --draft` / `--refresh` / `promote` / `validate --draft` with SHA-256 sidecar integrity, refuse-always shadowing safeguards, `NonoError::ActionRequired` typed variant, C FFI mapping, and advisory+strict `package_status.rs`.**

This plan absorbed upstream Luke Hinds commit `829c341a` (v0.47.1 — "add commands to manage profile drafts and check package status") into the nono fork using the D-20 manual-replay strategy mandated by the cherry-pick probe: 533 LOC across 6 files with 2 structurally-divergent hunks made a clean cherry-pick impossible. The absorption closes REQ-PORT-CLOSURE-03 (all 5 acceptance criteria) and flips deferred item P34-DEFER-04b-2 (tracked since Phase 34 upstream sync) from open to closed. The three commits were sequenced so Commit 1 backs the `ActionRequired` resolve_via display string for the `--refresh` workflow before Commit 2 introduces the error variant, and Commit 3 adds the validation surface, package status module, and all tests and docs.

Fork divergences required deliberate design decisions beyond upstream: the sidecar `.base` file approach (vs draft-internal hash field) keeps draft JSON schema-valid for `validate --draft --strict`; the refuse-always shadowing policy (vs upstream's `--force` escape hatch) is a security hardening per CLAUDE.md §Fail Secure; and the advisory-by-default posture for `package_status.rs` matches upstream 829c341a best-effort intent while providing a strict opt-in (`NONO_REQUIRE_PACK_STATUS=1`) that satisfies CLAUDE.md §Permission Scope for official pack targets. The `reserved_profile_source` function inlines `would_shadow_builtin` logic (D-36.5-C2-DEV1) because `profile_save_runtime.rs` is Windows-gated, avoiding cfg-propagation complexity across the call chain.

## Decisions Made

**D-36.5-C1 — D-20 manual-replay for all 3 commits (honored)**
Cherry-pick probe measured 533 LOC across 6 files with 2 structurally-divergent hunks. Fork divergences (WR-01 fail-closed in `profile/mod.rs`, Linux-cfg-gated `pre_create_landlock_profiles_dir` in `profile_runtime.rs`, struct-vs-tuple `ActionRequired`, sidecar-vs-internal hash, Unix-only `O_NOFOLLOW`, missing `find_pack_store_profile` analog) meant even conflict-free apply would not produce a working tree. Commit bodies cite `829c341a` as design source; no D-19 `Upstream-commit:` trailer.

**D-36.5-C2 — Sidecar `.base` file approach (honored)**
`<name>.base` stores SHA-256 of canonical profile bytes alongside the draft JSON. Draft JSON remains schema-valid so `validate --draft --strict` does not encounter a synthetic top-level hash field.

**D-36.5-D3 — Refuse-always shadowing with no `--force` (honored)**
`cmd_promote` refuses if the canonical name matches a built-in profile OR a package-managed profile — no escape hatch. Upstream permits `--force` for built-ins; this fork diverges per CLAUDE.md §Fail Secure. Documented in `profiles-groups.mdx`.

**D-36.5-C4 — 8-step close gate (honored)**
All 8 close-gate steps executed. Gates 1-5 green (see Verification Results). Gates 6-8 documented-skipped: no `exec_strategy`, WFP, or `learn` surface was touched by this plan.

**D-36.5-C2-DEV1 — `reserved_profile_source` inlines `would_shadow_builtin` (in-flight deviation)**
Deviation: the plan specified reusing `profile_save_runtime.rs::would_shadow_builtin` via a public call. At implementation time, `profile_save_runtime.rs` is Windows-gated (`#[cfg(target_os = "windows")]` module) making a cross-platform call from `profile_cmd.rs` require cfg-propagation that introduces more complexity than it prevents. Resolution: shadow detection logic inlined in `reserved_profile_source` in `profile_cmd.rs` (single semantic source). `would_shadow_builtin` was still promoted to `pub(crate)` in Commit 2 for potential future reuse on Windows paths.

**ActionRequired C FFI mapping**
`NonoError::ActionRequired { .. }` maps to `NonoErrorCode::ErrConfigParse (-9)`. No new error code value added; all language bindings unaffected.

**`package_status` advisory-by-default posture**
Matches upstream 829c341a best-effort intent: lockfile-load failure, missing-installed-version, and network-fetch failure all return `Ok(())`. Strict mode opt-in via `NONO_REQUIRE_PACK_STATUS=1` applies only to `OFFICIAL_PACK_STATUS_TARGETS`. Display-only `unwrap_or_default` on `installed_status`/`latest_version` stderr rendering retained (not a security boundary).

## Deviations from Plan

### D-36.5-C2-DEV1 — `reserved_profile_source` inlines `would_shadow_builtin`

- **Found during:** Commit 2 (cmd_promote + shadowing safeguards)
- **Issue:** Plan specified calling `profile_save_runtime.rs::would_shadow_builtin` from `profile_cmd.rs`. `profile_save_runtime.rs` is `#[cfg(target_os = "windows")]`-gated; importing from it in cross-platform `profile_cmd.rs` requires either cfg duplication or a module re-export bridge that adds more indirection than the code saves.
- **Fix:** Shadow detection logic inlined in `reserved_profile_source` in `profile_cmd.rs`. `would_shadow_builtin` still promoted to `pub(crate)` as planned, preserving future Windows reuse path.
- **Files modified:** `crates/nono-cli/src/profile_cmd.rs`, `crates/nono-cli/src/profile_save_runtime.rs`
- **Verification:** Cross-target clippy (Linux + macOS) clean; shadow refusal tests pass.
- **Committed in:** `857e5770` (Commit 2/3)

### rustfmt-drift handling (C1 / C2 / C3 orchestrator stash pattern)

Pre-existing rustfmt drift in `crates/nono-cli/src/sandbox_state.rs` and `crates/nono-cli/src/wiring.rs` (unrelated to this plan) was present before Commit 1. The orchestrator ran `cargo fmt --all` before each of the 3 commits to satisfy the `make fmt-check` CI gate, then stashed the drift files around each plan commit to keep plan commits clean.
