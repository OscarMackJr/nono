<objective>
Implement the D-09 unifying first-run-in-user-context provisioner: a SINGLE idempotent code path that
does the three things the SYSTEM MSI cannot — (1) create a user-owned `WRITE_OWNER` scratch dir under
`%LOCALAPPDATA%\nono\` so the R-B3 guard passes (DEPLOY-03), (2) import the POC cert into
`CurrentUser\Root` (DEPLOY-05), (3) set+persist `NODE_EXTRA_CA_CERTS` to the MSI-staged PEM at
`%PROGRAMDATA%\nono\nono-poc-root.pem` so confined Node engines trust the proxy (DEPLOY-05). Factor the
cert-import logic as testable Rust with a SINGLE store-list source of truth (`cert_trust`), expose a
`nono setup --trust-root <cer>` verb that the Plan 01 MSI custom action invokes (so the
`Root`+`TrustedPublisher` list lives in Rust, not duplicated in a PowerShell here-string), and reuse
that module's per-user entry point from this provisioner.

Purpose: The MSI lays down machine-global state only (incl. the PEM and the DER cert + the cert CA that
shells `nono setup --trust-root`); everything user-owned is auto-provisioned at first run in the user's
own token (Pitfall 4 / D-08). Single idempotency check, single failure surface that `nono health`
(Plan 03) reads, single Rust store-list. Library/CLI boundary is sacrosanct — all logic lands in
nono-cli; the `nono` library stays policy-free and is consumed only for its R-B3 ownership primitives.
Output: `cert_trust.rs`, `provision_windows.rs`, the `setup --trust-root` clap flag + setup-runtime arm,
a non-fatal hook in `command_runtime::run_sandbox`, and the module declarations in `lib.rs`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| invoking user process → filesystem ACLs | The provisioner runs in the user's token and mutates ownership/DACLs on a new scratch dir. |
| provisioner → CurrentUser\Root trust store | A user-context process imports a trust-root cert into the per-user store. |
| provisioner → confined Node child (NODE_EXTRA_CA_CERTS) | The provisioner injects a CA-bundle path consumed by the sandboxed Node engine. |
| MSI deferred CA (LocalSystem) → `nono setup --trust-root` → machine trust stores | The MSI invokes the CLI verb under SYSTEM to import the root cert into Root+TrustedPublisher. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-82-10 | Elevation of Privilege | scratch dir created with overly broad DACL | mitigate | Set ownership to ONLY the invoking user via `icacls /setowner *<SID>` and verify with `path_is_owned_by_current_user`; grant write to that SID only (R-B3 primitives) — no Everyone/Authenticated-Users grant. |
| T-82-11 | Tampering (TOCTOU) | symlink/dir-swap between create and ownership-set on %LOCALAPPDATA%\nono | mitigate | %LOCALAPPDATA% is user-owned and not world-writable (unlike %TEMP% shared scratch); create-then-verify-ownership closes the window; on `Ok(false)` from the ownership check, record degraded rather than trusting the dir. Path-component comparison only (CLAUDE.md), never string starts_with. |
| T-82-12 | Spoofing | wrong cert imported into CurrentUser\Root | mitigate | Cert path resolved deterministically from INSTALLFOLDER / `current_exe().parent()` (Admins-only Program Files staging by the MSI), not a user-writable or argument-supplied path; idempotency probe matches by thumbprint. |
| T-82-13 | Information Disclosure | NODE_EXTRA_CA_CERTS points at a swappable path → MITM of confined Node TLS | mitigate | Point at the MSI-staged PEM under `%PROGRAMDATA%\nono\` (Admins-only writable), not a user-writable copy; the value is a CA bundle (public cert), not a secret. The setx persistence writes only the path, not the cert. |
| T-82-14 | Denial of Service | provisioning failure aborts every `nono run` | mitigate | All steps are NON-FATAL by design (DEPLOY-03/06): failures record degraded status for `nono health` and the run continues — a broken provisioner cannot brick the agent. |
