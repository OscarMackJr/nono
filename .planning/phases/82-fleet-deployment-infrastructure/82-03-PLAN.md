<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SCCM/Intune compliance script → `nono health` JSON/exit | Fleet management consumes the verdict to decide remediate-vs-reinstall; the JSON must not leak sensitive host detail. |
| `nono health` → SCM / HKLM / cert stores | A read-only diagnostic queries privileged subsystems. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-82-20 | Information Disclosure | health JSON leaks full user paths / sensitive scratch location into SCCM logs | mitigate | Report scratch/cert/PATH state as booleans/enums (present/owned/missing), NOT raw absolute user paths; the JSON carries verdict + per-group status, not file contents or full per-user paths. (Aligns with the Phase 84 SecurityEvent no-raw-path principle, applied early.) |
| T-82-21 | Tampering | health command mutates state while "diagnosing" | mitigate | Command is strictly read-only: SCM query, `reg query`, cert presence probe, PATH read, marker read — NO create/write/addstore/setowner calls. Acceptance criterion asserts no mutation calls. |
| T-82-22 | Spoofing | fail-OPEN: a probe error reported as "healthy" | mitigate | A subsystem probe error maps to Degraded/Broken (fail-secure for the verdict), never silently Healthy; `unwrap_or_default()` is forbidden (clippy + CLAUDE.md). An unreadable HKLM sentinel → "unreadable" (not "not-configured"). |
| T-82-23 | Denial of Service | a hung SCM/reg subprocess wedges `nono health` | accept | Read-only subprocess probes against local SCM/registry; low risk on a healthy host. A timeout wrapper is a future hardening, not Phase 82-blocking. |
| T-82-SC | Tampering | npm/pip/cargo installs | accept | This plan adds NO new crates (reuses serde_json + std + platform.rs subprocess conventions). No package-legitimacy gate required. |
</threat_model>
