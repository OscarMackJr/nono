<objective>
Gap-closure for F-62-UAT-05 (root-caused + DESIGNED in debug session wfp-broker-token-no-sid; status design_complete).
After 62-05..62-09 the WFP block filter installs cleanly (FwpmFilterAdd0 returns 0), but in live Win11 UAT the confined
child STILL reached the network: `curl` printed the external IP and exited 0. Root cause: the filter scopes to the
connection's user token via FWPM_CONDITION_ALE_USER_ID with security descriptor `D:(A;;CC;;;<session_sid>)`, but the
broker arm (BrokerLaunchNoPty, selected by profile.windows_low_il_broker=true e.g. claude-code) builds the child token
via `nono::create_low_integrity_primary_token()` which takes NO SID — so the child token does not carry session_sid and
the filter matches NOTHING. (dacl_guard.rs L32-34 documents this: the synthetic SID is on no child token for the
BrokerLaunch*/LowIlPrimary arms; only WriteRestricted carries it — and WriteRestricted breaks child startup.)

Fix (debug D1/D3/D4/D5): inject session_sid into the broker's Low-IL primary token as the single RESTRICTING SID under
WRITE_RESTRICTED — the proven WFP-matchable shape (restricted_token.rs is the working reference) — via a NEW library fn
`create_low_integrity_primary_token_with_sid`, then thread `--session-sid` through launch.rs (no-PTY broker_args) → broker
argv parser → broker token build, with a FAIL-CLOSED contract at all three sites.

SCOPE: this plan closes the SC1 enforcement-MATCH gap only. The leave-nothing uninstall purge it relates to (persistent
WFP objects from 62-09) is plan 62-11, scheduled before SC4. The ONE empirical unknown — does `curl`/`powershell` start
under WRITE_RESTRICTED+Low-IL on the broker path? — is validated by the live Win11 SC1 re-run after this plan, NOT here.

Output: broker child token carries session_sid; WFP ALE_USER_ID filter matches the confined child; outbound is blocked.
