<objective>
Gap-closure for F-62-UAT-03: the second independent cause of FwpmFilterAdd0 returning RPC_X_BAD_STUB_DATA
(win32 1783) on the --block-net path. add_policy_filter sets the ALE_USER_ID condition's
FWP_CONDITION_VALUE0_0.sd to a RAW security descriptor (`security_descriptor as *mut _`), but windows-sys 0.59
types that union member as `*mut FWP_BYTE_BLOB`. WFP requires FWP_SECURITY_DESCRIPTOR_TYPE values to be an
FWP_BYTE_BLOB whose `data` points to a SELF-RELATIVE security descriptor and `size` is its length. Passing a raw
SD makes the BFE RPC stub marshal the SD bytes as a bogus FWP_BYTE_BLOB → RPC_X_BAD_STUB_DATA (1783). For a
supervised --block-net run the session_sid branch always installs this SD condition (install_wfp_policy_filters
L1436-1442), so the path is always hit. This stacks on top of the 62-07 null-name fix (both were 1783 causes on
the same path).

This plan wraps the self-relative SD in a lifetime-correct FWP_BYTE_BLOB. No other field changes — a manual
value-shape re-audit confirmed the SD condition is the only remaining malformed value.

Output: add_policy_filter builds an FWP_BYTE_BLOB { size: GetSecurityDescriptorLength(sd), data: sd as *mut u8 }
and points the ALE_USER_ID condition's sd member at that blob.
</objective>
