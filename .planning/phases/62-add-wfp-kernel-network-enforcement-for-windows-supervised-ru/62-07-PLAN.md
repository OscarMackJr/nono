<objective>
Gap-closure for F-62-UAT-02 (root-caused in debug session wfp-filter-add-1783). The WFP block filter install
in nono-wfp-service.rs `add_policy_filter` passed `filter.displayData.name = null_mut()`. FWPM_DISPLAY_DATA0.name
is a REQUIRED PWSTR; FwpmFilterAdd0 marshals it over RPC to the BFE engine, and a NULL required string fails the
RPC stub with RPC_X_BAD_STUB_DATA (win32 1783) — the exact live-UAT error. The already-working create_nono_sublayer
sets a non-null name and FwpmSubLayerAdd0 succeeds; this fix applies the same pattern to the filter.

The debug field audit (full 7-point FWPM review) classified this SMALL: the null name is the ONLY defect; weight,
all four condition value-type/field-key pairings, numFilterConditions, layer/sublayer keys, the action union, and
pointer liveness of app_id_blob/security_descriptor/conditions/weight_value are all already correct. So this plan
changes exactly one field (the display name) and nothing else.

Purpose: unblock REQ-WFP-01 SC1 — the service can actually install the kernel WFP block filter instead of failing
closed at FwpmFilterAdd0.

Output: add_policy_filter builds a non-null UTF-16 display-name buffer (lifetime-correct) and assigns it to
filter.displayData.name.
</objective>
