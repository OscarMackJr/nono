<objective>
Gap-closure for F-62-UAT-01 (root-caused in debug session wfp-driver-gate-placeholder). The client-side
readiness gate build_wfp_probe_status made the out-of-scope kernel driver (nono-wfp-driver) an unconditional
precondition of WfpProbeStatus::Ready. Since Ready is the SOLE status that triggers the activation IPC in
install_wfp_network_backend_with_runner (L1625) — the IPC that reaches the service's real, fully-wired
FwpmFilterAdd0 enforcement path — and D-05 deliberately leaves the kernel driver unregistered, every
network.block run failed closed at the driver gate before enforcement could activate. This directly
contradicted D-05 ("service-only — the user-mode nono-wfp-service IS the kernel enforcement; the kernel
driver is out of scope").

This plan removes the kernel-driver requirement from the path to Ready so BFE-running + nono-wfp-service-running
is sufficient, preserving every other fail-secure prerequisite. It also retires the stale SERVICE-side
"enforcement not implemented / placeholder service host" strings (the service enforcement IS implemented), while
keeping the DRIVER-side placeholder strings (the kernel driver genuinely remains out of scope).

Purpose: unblock REQ-WFP-01 SC1 — a machine-MSI supervised `nono run --block-net` reaches the service's WFP
filter activation out of the box instead of failing closed at a driver gate that contradicts the phase's own design.

Output: build_wfp_probe_status (and its run-path caller) no longer gate on the kernel driver; stale service
strings retired; a new unit test proving Ready service-only.
</objective>
