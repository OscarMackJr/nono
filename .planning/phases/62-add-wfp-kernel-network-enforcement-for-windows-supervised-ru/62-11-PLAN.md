<objective>
Follow-up to 62-09 (renumbered from the reserved "62-10"). 62-09 made the WFP session PERSISTENT to fix
FWP_E_WRONG_SESSION — but persistent objects survive engine-handle close AND survive `msiexec /x`. The runtime cleanup
(startup orphan sweep + per-request remove-by-key) handles a normal run, but a plain uninstall now leaves the persistent
sublayer (NONO_SUBLAYER_GUID) and any crash-surviving BLOCK filters in the BFE object store forever. That violates the
leave-nothing contract (REQ-WFP-01 SC4 clean uninstall / SC5; REQ-DRN-01 principle).

Fix: add a one-shot `--purge-wfp-objects` mode to the service binary that deletes every filter under NONO_SUBLAYER_GUID
(reusing the startup-sweep enumeration) and then deletes the sublayer itself (FwpmSubLayerDeleteByKey0). Wire the existing
`nono.exe setup --uninstall-wfp` path (already a LocalSystem deferred custom action, Before=RemoveFiles) to run the purge
before it stops+deletes the service. FAIL-OPEN: a purge failure is reported but never blocks uninstall.

SCOPE: cleanup only. No WiX change (the custom action already calls setup --uninstall-wfp). No runtime-path change. This
plan closes the SC4/SC5 leave-nothing gap that 62-09 deliberately deferred; it depends on 62-10 only for sequencing
(broker SC1 fix lands first).

Output: uninstall removes the persistent nono sublayer + filters; reinstall starts from a clean object store.
</objective>
