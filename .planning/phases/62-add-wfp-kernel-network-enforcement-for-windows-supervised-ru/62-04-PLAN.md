<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: SC1 + SC5 — Out-of-box enforced block (no prior start-wfp-service)</name>
  <what-built>
    Machine MSI with Start="auto" ServiceInstall and PIPE_SDDL with IU ACE is installed.
    The SCM will have started the service during install (ServiceControl Start="install").
    SC1: Without any prior `nono setup --start-wfp-service`, a non-elevated supervised run
    with network.block:true should deny the confined child's outbound TCP.
    SC5: The confined child itself (Low-IL) does NOT connect to the WFP pipe — only nono.exe
    (Medium IL) does. Verify that the WFP block is due to kernel filters, not pipe denial.
  </what-built>
  <how-to-verify>
    Step 1 (elevated terminal): Install the rebuilt machine MSI:
      msiexec /i path\to\nono-machine.msi /quiet

    Step 2: Verify the service is running (started by WiX ServiceControl during install):
      sc query nono-wfp-service
      Expected STATE: 4 RUNNING

    Step 3: Verify service start type is AUTO (NOT demand):
      sc qc nono-wfp-service
      Expected START_TYPE: 2 AUTO_START
      If START_TYPE is 3 DEMAND_START, STOP — the WiX change did not land in the MSI.

    Step 4 (NON-ELEVATED PowerShell — open a new non-admin window): Run a confined network test
    using the runner profile with network.block:true. The runner profile must set network.block:true
    (check the profile or pass a temporary override). Example command to confirm network is blocked:
      nono run --profile runner -- powershell -Command "try { (Invoke-WebRequest https://api.ipify.org -TimeoutSec 5).Content } catch { 'BLOCKED: ' + $_.Exception.Message }"
    Expected output: "BLOCKED:" prefix or a connection-refused error, NOT the external IP address.

    Step 5 (SC5 verification): Check that the above block is a WFP kernel filter, not a pipe
    access denial. If Step 4 showed "Access is denied" connecting to the control pipe, STOP —
    that means the PIPE_SDDL IU change did not land or the MSI binary is stale.
    The correct failure from the confined child is a TCP connection timeout/reset, not a pipe error.
    nono.exe output should say "WFP enforcement active" (or similar), NOT "Access is denied".

    Step 6 (verify explicitly-allowed ports pass): If the runner profile allows an explicit port
    (e.g., localhost:3000 for development), confirm that port is still reachable from a non-confined
    process (nono-wfp-service enforces only the confined session's SID).

    Expected outcomes:
    - Outbound TCP to external hosts: BLOCKED (WSAETIMEDOUT or WSAECONNREFUSED from confined child)
    - nono.exe console: no "Access is denied" pipe error; enforcement activated without manual setup step
    - Explicitly-allowed ports: accessible (not collateral damage)
  </how-to-verify>
  <resume-signal>Type "sc1-pass" if confined outbound is blocked and pipe access was clean, or describe the failure mode (include the exact error output).</resume-signal>
</task>

    Expected outcomes:
    - Non-elevated run with service stopped: exits non-zero with message containing "nono setup --start-wfp-service" and "elevated"
    - NO "hello" output (not proceeded unenforced)
    - After sc start (elevated), non-elevated run: WFP enforcement active
