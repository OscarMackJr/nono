# Milestone v2.2: Windows/macOS Parity Sweep

**Status:** ✅ SHIPPED 2026-04-29
**Phases:** 22 – 24
**Total Plans:** 9 plans across 3 phases
**Branch:** `windows-squash` (continuing from v2.1)

## Overview

Close the Windows-vs-macOS drift opened by upstream `always-further/nono` shipping v0.38.0, v0.39.0, v0.40.0, and v0.40.1 without Windows ports. Land profile struct alignment, policy tightening, the `nono package` subcommand tree, OAuth2 client-credentials proxy injection, audit integrity + DSSE attestation + Windows Authenticode exec-identity recording, and Windows AIPC broker audit emissions — preserving v2.1 Phase 19 CLEAN-04 invariants byte-identically through the `prune` → `session cleanup` rename. Install drift-prevention tooling (`make check-upstream-drift` twin scripts + GSD upstream-sync template) so v0.41+ becomes a maintenance task instead of a milestone-scale sync. Delivered against 21 v2.2 requirements (PROF, POLY, PKG, OAUTH, AUD, DRIFT) with 0 orphans.

**Core value:** Every nono command that works on Linux/macOS works on Windows with equivalent security guarantees, or is explicitly documented as intentionally unsupported with a clear rationale.

**Pre-milestone (not a v2.2 phase):** `windows-squash` → `main` merge tracked as a separate quick task (still pending on PR-583 maintainer response per quick-260428-rsu re-deferral).

## Milestone Summary

**Decimal Phases:** None inserted in v2.2.

**Plan Splits:**

- **Plan 22-05 → 22-05a + 22-05b** (split at CONTEXT STOP trigger #3 on upstream cherry-pick `4f9552ec`). The `prune` → `session cleanup` rename touches `rollback_runtime.rs` / `supervised_runtime.rs` / `exec_strategy.rs`, all heavily forked on `windows-squash`. Landing the rename atomically with the audit-core upstream port would have made the cherry-pick chain irreversible mid-plan. T-22-05-04 ABSOLUTE STOP guard required CLEAN-04 invariants byte-identical AFTER every source-code commit, so the split honored the gate. Formal `applied_labels_guard::audit_flush_before_drop` regression test (83 LOC) installed as future-regression guard.

**Key Decisions:**

- **Phase 22 PKG flat-shape over nested `nono package` parent** — Upstream landed package operations as 5 flat top-level subcommands (`pull` / `remove` / `update` / `search` / `list`); the fork mirrors that shape verbatim per D-19 cross-phase byte-identical preservation. Verified during Phase 22 UAT (Test 7 spec was wrong; 7b confirmed actual ship-shape).
- **Phase 22 audit-integrity verification upgraded to cryptographic DSSE (HG-01-H)** — Initial 22-05a Plan implementation only verified the *structural shape* of `audit-attestation.bundle`. HG-01-H reviewer caught that this would silently accept a forged bundle whose signature was wrong. Cryptographic DSSE verification fail-closes on any signature mismatch (commit `cffb43b1`).
- **Phase 22 Authenticode discriminant-only over chain-walker subject extraction (Decision 4 fallback)** — `windows-sys 0.59` does not expose `WTHelperProvDataFromStateData` / `WTHelperGetProvSignerFromChain` without `Win32_Security_Cryptography_Catalog` + `Win32_Security_Cryptography_Sip` features. Recording `Valid` / `Unsigned` / `InvalidSignature{hresult}` discriminant captures the security-relevant trust-decision boundary; subject + thumbprint are nice-to-have audit metadata deferred to v2.3.
- **Phase 23 layer-2 deviation authorized by plan Step 7** — `tests/aipc_handle_brokering_integration.rs` calls lower-level broker functions and CANNOT call `handle_windows_supervisor_message` (which is `pub(super)`); the layer-1 multi-kind test in `capability_handler_tests` provides the authorized substitute coverage.
- **POLY-02 mutex carve-out (CL-01-M)** — `--rollback` ↔ `--no-audit` is the only POLY-02 mutex; `--no-audit-integrity` (cryptographic ledger only) is orthogonal to filesystem-snapshot rollback and now allowed alongside `--rollback`.
