# Quick Task 260522-v14: WRITE_OWNER Pre-Flight Check Summary

## One-liner

The Windows mandatory-integrity-label backend now runs a `GetEffectiveRightsFromAclW` pre-flight inside `try_set_mandatory_label` and surfaces a precise directive error naming `WRITE_OWNER` + recommending `%USERPROFILE%\nono-poc` / `%TEMP%\nono-poc` (or an explicit `icacls /grant` local override) when the calling user lacks WRITE_OWNER on the target ŌĆö replacing the misleading "writable...NTFS..." catch-all that was confusing the POC pilot user on `C:\poc\temp`.

## What was built

### Production code

**`crates/nono/src/sandbox/windows.rs` (line 1050) ŌĆö new public function `path_has_write_owner`**

```rust
pub fn path_has_write_owner(path: &Path) -> Result<bool>
```

Returns `Ok(true)` if the current-user SID has `WRITE_OWNER` (0x00080000) in its effective rights against the path's DACL; `Ok(false)` otherwise. Fail-closed: any FFI failure (DACL read, token open, token-user query, effective-rights query) returns `Err(NonoError::LabelApplyFailed)` carrying the underlying Win32 error code. Callers MUST NOT treat `Err` as "WRITE_OWNER present" ŌĆö that would be a silent permission widening (T-v14-01).

**SAFETY invariants** documented inline on every `unsafe { ... }` block:
- `GetNamedSecurityInfoW(DACL_SECURITY_INFORMATION)` ŌĆö DACL pointer lives inside the returned SD; freed via `OwnedSecurityDescriptor` RAII guard.
- `OpenProcessToken(TOKEN_QUERY)` ŌĆö uses pseudo-handle from `GetCurrentProcess()`; closed via internal `OwnedTokenHandle` RAII guard.
- Two-call `GetTokenInformation(TokenUser)` size-probe pattern (matches `path_is_owned_by_current_user`).
- `TRUSTEE_W` built via struct literal (windows-sys 0.59 exposes the struct directly; no `BuildTrusteeWithSidW` FFI hop needed).
- `GetEffectiveRightsFromAclW` ŌĆö `dacl` lives in `_sd_guard`, `trustee` holds a SID pointer into the TOKEN_USER buffer (owned by the stack frame).

**Null-DACL special case:** A null DACL allows everyone every access (effectively WRITE_OWNER present) ŌĆö function returns `Ok(true)` in that case. Documented inline.

**Imports added** (lines 22-34 of `windows.rs`):
- `Win32::Security::Authorization`: `GetEffectiveRightsFromAclW`, `NO_MULTIPLE_TRUSTEE`, `TRUSTEE_IS_SID`, `TRUSTEE_IS_USER`, `TRUSTEE_W`
- `Win32::Security`: `DACL_SECURITY_INFORMATION`
- `Win32::Storage::FileSystem`: `WRITE_OWNER` (= 524288u32 = 0x00080000)

**`crates/nono/src/sandbox/windows.rs` (line 695) ŌĆö pre-flight wiring in `try_set_mandatory_label`**

The check sits at the TOP of the function body, immediately after the local `use windows_sys::Win32::Foundation::{ERROR_ACCESS_DENIED, ...}` import block, before the SDDL construction:

```rust
if !path_has_write_owner(path)? {
    return Err(NonoError::LabelApplyFailed {
        path: path.to_path_buf(),
        hresult: ERROR_ACCESS_DENIED,
        hint: format!(
            "The current user lacks WRITE_OWNER (0x00080000) on this path. \
             Mandatory integrity labels require WRITE_OWNER, which is NOT implicit for path owners. \
             User-created subdirectories of a drive root (e.g. C:\\poc\\) inherit the default C:\\ ACL, \
             which grants only `Authenticated Users: Modify` ŌĆö WRITE_OWNER is missing. \
             Recommended: run nono from a working directory under your user profile \
             (e.g. %USERPROFILE%\\nono-poc or %TEMP%\\nono-poc). \
             Local override: grant FullControl on the current path via \
             `icacls {} /grant <user>:(OI)(CI)F` (this widens the DACL beyond \
             default inheritance ŌĆö explicit user choice, not a default).",
            path.display(),
        ),
    });
}
```

**Existing catch-all preserved unchanged:** the `ERROR_ACCESS_DENIED | ERROR_INVALID_FUNCTION | ERROR_NOT_SUPPORTED` branch at the bottom of `try_set_mandatory_label` still emits the "Ensure the target file is writable by the current user and is on NTFS..." text ŌĆö that hint remains correct for the genuine-mystery case (EDR, FS driver quirks, WRITE_OWNER-present-but-SetNamedSecurityInfoW-still-fails).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug, test scaffolding only] Synthetic DACL false positive via GetEffectiveRightsFromAclW group walking**

- **Found during:** Task 2, first-pass execution of `try_set_mandatory_label_surfaces_directive_when_write_owner_missing`.
- **Issue:** The plan-suggested SDDL `D:PAI(A;OICI;FA;;;BA)(A;OICI;0x1301BF;;;AU)` (BUILTIN\\Administrators FullControl + Authenticated Users Modify) gave a false positive on this dev host. `GetEffectiveRightsFromAclW` walks the trustee's group memberships from the **full** token, not the UAC-filtered token. The test runner (and the dev host's normal interactive user) is a member of `BUILTIN\\Administrators` in their full token, so the function returned FullControl for the user SID ŌĆö pre-flight reported `Ok(true)`, test fell through to the OLD catch-all NTFS hint, assertion failed expecting `WRITE_OWNER` in the message.
- **Fix:** Look up the current-user SID dynamically (via `OpenProcessToken` + `GetTokenInformation(TokenUser)` + `ConvertSidToStringSidW`), build the SDDL with the explicit user SID as the trustee with mask `0x1301BF` only. This structurally guarantees `WRITE_OWNER` is absent regardless of which groups the user belongs to in the full vs filtered token.
- **Why this is test-scaffolding only (not a production-code concern):**
  - The production path probes the path's REAL DACL (typically the inherited default `C:\` ACL on user-created drive-root subdirs).
  - On `C:\poc\temp`, the default ACL grants `BUILTIN\\Users: ReadAndExecute` and `Authenticated Users: AppendData/CreateSubdirectory` style ACEs ŌĆö neither includes WRITE_OWNER.
  - The user's report confirmed the FFI call ACTUALLY fails with `ERROR_ACCESS_DENIED` under the filtered token on their machine, meaning the real-world DACL does NOT include any ACE giving the user (via any group membership) WRITE_OWNER.
  - The synthesized-DACL test was the only place the false positive could surface, because we control the exact ACL composition and chose to include BUILTIN\\Administrators (for cleanup). Using the explicit user SID is structurally tighter regardless.
- **Files modified:** `crates/nono/src/sandbox/windows.rs` (negative test only).
- **Commit:** `27e2db4b` (Task 2).

**Caveat for the POC user (post-merge surfacing risk, not blocking ship):**
If the POC user's `C:\poc\temp` path turns out to have an inherited ACE that DOES grant their user (via some group like `BUILTIN\\Administrators`) WRITE_OWNER in the full token, the pre-flight check would return `Ok(true)` and the user would still see the OLD NTFS hint after the actual filtered-token call fails. The user's own debug session report (`.planning/debug/il-label-apply-access-denied.md`) suggests this is NOT the case on their machine (the failure pattern they describe is the FFI returning `ERROR_ACCESS_DENIED`, which means the OS access check rejected the call ŌĆö meaning the user genuinely lacks WRITE_OWNER under both the filtered AND full token on that path). If post-merge UAT shows otherwise, the architectural choice between `GetEffectiveRightsFromAclW` (LOCKED #2) and `AccessCheck` (rejected by debugger) would need to be revisited ŌĆö but that is a Rule 4 decision for next iteration, not part of this fix.
