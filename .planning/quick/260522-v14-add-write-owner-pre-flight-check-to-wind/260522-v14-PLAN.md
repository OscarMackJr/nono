    /// Returns `Ok(true)` if the current-user SID has `WRITE_OWNER` in its
    /// effective rights against `path`'s DACL; `Ok(false)` otherwise.
    ///
    /// # Why
    ///
    /// `SetNamedSecurityInfoW(LABEL_SECURITY_INFORMATION)` requires the caller
    /// to hold `WRITE_OWNER` (0x00080000) on the target object. Owner status
    /// grants implicit `WRITE_DAC` + `READ_CONTROL` per the NT security model
    /// but does NOT imply `WRITE_OWNER` — that bit must come from a DACL ACE.
    /// The default `C:\` root ACL inherited by user-created drive-root
    /// subdirectories (e.g. `C:\poc\temp`) caps users at `Modify` via
    /// `Authenticated Users`, which is missing the `WRITE_OWNER` bit. Probing
    /// the effective rights via `GetEffectiveRightsFromAclW` before the
    /// label-apply call lets us surface a precise directive error instead of
    /// the misleading "writable...NTFS..." catch-all.
    ///
    /// # Errors
    ///
    /// Fail-closed: any FFI failure (DACL read, token open, token-user query,
    /// effective-rights query) returns `NonoError::LabelApplyFailed` carrying
    /// the underlying Win32 error code. Callers MUST NOT treat this `Err` as
    /// "WRITE_OWNER present" — that would be a silent permission widening.
    pub fn path_has_write_owner(path: &Path) -> Result<bool> {
        // 1. GetNamedSecurityInfoW(DACL_SECURITY_INFORMATION) — read the path's DACL.
        //    Wrap the returned PSECURITY_DESCRIPTOR in `OwnedSecurityDescriptor`.
        //    On non-zero status, return Err(NonoError::LabelApplyFailed { ..., hint: "GetNamedSecurityInfoW(DACL_...) returned 0x{status:08X} ..." }).
        // 2. OpenProcessToken(GetCurrentProcess(), TOKEN_QUERY, ...) — wrap in OwnedTokenHandle.
        // 3. GetTokenInformation(TokenUser) two-call pattern → extract user SID pointer (PSID into local buffer).
        // 4. Build TRUSTEE_W via direct struct initialization (NOT BuildTrusteeWithSidW — that does the same thing
        //    via FFI and the struct is publicly exposed by windows-sys):
        //       TRUSTEE_W {
        //           pMultipleTrustee: std::ptr::null_mut(),
        //           MultipleTrusteeOperation: MULTIPLE_TRUSTEE_OPERATION(NO_MULTIPLE_TRUSTEE),  // 0
        //           TrusteeForm: TRUSTEE_FORM_SID,    // 0
        //           TrusteeType: TRUSTEE_TYPE_USER,   // 1
        //           ptstrName: user_sid as *mut _,
        //       }
        //    Verify the EXACT field names + numeric variants against `windows-sys = "0.59"` by reading the
        //    crate's types in `~/.cargo/registry/src/.../windows-sys-0.59.0/src/Windows/Win32/Security/Authorization/mod.rs`
        //    BEFORE writing the struct literal — windows-sys field shapes shift between minor versions.
        //    If `MULTIPLE_TRUSTEE_OPERATION` and `TRUSTEE_FORM` are newtype-wrapped enums in 0.59, use the
        //    corresponding `.0` discriminants or named constants (windows-sys 0.59 typically exports them as
        //    `MULTIPLE_TRUSTEE_OPERATION(0)` style — confirm before committing).
        // 5. let mut access_mask: u32 = 0;
        //    let status = GetEffectiveRightsFromAclW(dacl_ptr, &trustee, &mut access_mask);
        //    On non-zero, return Err(...). On zero, Ok((access_mask & WRITE_OWNER) != 0).

        //    Worst case, define a local `const WRITE_OWNER: u32 = 0x0008_0000;` with a comment citing
        //    `winnt.h` if neither path exports it in 0.59 — but check first; the constant exists in 0.59 somewhere.
    }

    ### Working directory choice (Windows)

    On Windows, `nono run` and `nono shell` apply a mandatory integrity label (Low-IL
    SACL) to the working directory before spawning the child process. The Win32 call
    `SetNamedSecurityInfoW(LABEL_SECURITY_INFORMATION)` requires the calling user to
    hold `WRITE_OWNER` (0x00080000) on the target — a right that is NOT implicit for
    path owners, despite owner status granting implicit `WRITE_DAC` and `READ_CONTROL`.

    **Failure mode:** User-created subdirectories of a drive root (e.g. `C:\poc\temp`,
    `C:\projects\foo`) inherit the default `C:\` ACL, which grants `Authenticated Users:
    Modify` — a 14-bit mask (`0x1301BF`) that does NOT include `WRITE_OWNER`. Running
    `nono run --allow . -- cmd /c echo hello` from such a path fails with a directive
    error naming `WRITE_OWNER` and pointing you to a supported working directory.

    **Recommended POC working directories** (both have explicit `<user>: FullControl`
    ACEs from profile creation, which include `WRITE_OWNER`):

    ```powershell
    # Option A — under your user profile (persistent)
    mkdir $env:USERPROFILE\nono-poc
    cd $env:USERPROFILE\nono-poc

    # Option B — under %TEMP% (transient)
    $poc = Join-Path $env:TEMP "nono-poc-$(Get-Random)"
    mkdir $poc | Out-Null
    cd $poc
    ```

    `WRITE_OWNER`) on the path:

    icacls C:\poc\temp /grant ${env:USERNAME}:(OI)(CI)F
    ```

    This is a one-shot setup step per directory — nono itself never mutates DACLs;
    the runtime fails closed if WRITE_OWNER is missing rather than silently widening.
