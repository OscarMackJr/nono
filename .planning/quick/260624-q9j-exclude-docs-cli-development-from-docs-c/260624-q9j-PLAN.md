# Quick Task 260624-q9j: Fix red `Docs Checks` (windows-win-1706-option-1-workstream.mdx)

## Original framing (from todo) — CORRECTED during investigation

The todo (and the originally-chosen disposition "exclude `docs/cli/development/**`
from the nav scan") rested on a **false premise**: that the file existed on disk
but was *missing from* `docs.json` nav, and that `docs/cli/development/` is
intentionally unpublished.

Verifying against production code (`.github/workflows/ci.yml` job `docs-checks`,
step "Verify docs nav pages exist") inverted the diagnosis:

- The CI check is **nav → file**: `jq '.navigation.groups[].pages[]' docs/docs.json`
  then `[[ ! -f "docs/${page}.mdx" ]] && exit 1`.
- `cli/development/windows-win-1706-option-1-workstream` is **already in nav**
  (`docs.json` line 104), next to its sibling `windows-win-1706-decision-package`.
- The `.mdx` exists on the local disk but was **never `git add -f`'d**
  (`docs/cli/development/` is gitignored, `.gitignore:19`), so it is absent from
  the git tree CI checks out → "Missing page from docs.json navigation".
- The dir **is** published: ~16 sibling dev-doc pages are force-tracked AND in nav.

So "exclude the dir from scan" would have dropped valid nav-integrity coverage for
~16 real published pages to paper over one untracked file.

## Decision (operator, corrected)

**Force-add the file to git** — it is already wired into nav, so it is meant to be
published. Smallest, most honest fix; matches author intent.
