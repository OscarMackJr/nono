v2 of the WRITE_OWNER fix. The v1 pre-flight (`path_has_write_owner` + `GetEffectiveRightsFromAclW`) is structurally unsound — `GetEffectiveRightsFromAclW` walks group memberships from the FullToken, but `SetNamedSecurityInfoW(LABEL_*)` runs the access check against the UAC-filtered token. Local admins on the POC user's machine can get `Ok(true)` from the pre-flight and STILL fail the actual apply, falling through to the misleading "writable...NTFS..." catch-all. The v1 deviation note (Rule 1, executor's own write-up) flagged this; locked design directive #2 confirms removing it.

Purpose:
- Drop the broken pre-flight and its FFI surface (one function, one import block subset).
- Emit the WRITE_OWNER directive hint on the existing `SetNamedSecurityInfoW(LABEL_*)` ACCESS_DENIED branch, gated by `path_is_owned_by_current_user` — which uses the SAME (filtered) token the apply call runs under, so its judgment matches reality.
- Keep both hint strings as separate branches: directive for the user-owned common case, catch-all for the genuine-mystery non-user-owned case (EDR, FS driver quirks, etc.).

Output:
- `crates/nono/src/sandbox/windows.rs` modified — net deletion (~200 lines: the `path_has_write_owner` function + its positive test) with ~15 lines added (the new branch logic in the ACCESS_DENIED arm + the renamed test header).
